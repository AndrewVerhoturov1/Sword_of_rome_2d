# Иерархическая система AI-разработки через планирование, Boss Orchestrator и B1-блоки

---

# Верхний слой системы: «подпроект»

## Почему нужен подпроект

В иерархической системе AI-разработки нужен уровень выше Boss Orchestrator и выше отдельных B1-блоков. Эту сущность удобно называть **подпроектом**.

Подпроект — это не весь репозиторий и не одна конкретная задача в стиле “поправить файл”. Это рабочая область внутри большого проекта, в которой живёт связанная цепочка планирования, блоков, чатов, отчётов, решений, внешних ревью и локальных действий.

Если смотреть сверху, репозиторий — это технический контейнер всего проекта. Но внутри репозитория человек обычно работает не “со всем проектом сразу”, а с отдельной большой частью:

- перестройка V3 workflow;
- развитие B1/Boss orchestration;
- внедрение token diagnostics;
- улучшение редактора карты;
- создание системы импорта V3 packages;
- разработка нового модуля правил;
- рефакторинг runtime-архитектуры;
- подготовка Obsidian-памяти;
- создание нового workflow слоя.

Каждая такая большая часть может жить несколько дней или недель. Внутри неё могут быть Planning Chat, Boss Orchestrator, несколько B1-блоков, много Kilo-задач, V1/V2/V3 запросы, решения, отчёты и корректировки плана.

Без отдельной сущности “подпроект” всё это начинает смешиваться в общей `.ai`-памяти репозитория. Через некоторое время становится трудно понять:

- к какой большой работе относится этот V1-ответ;
- какой B1-блок закрывал этот Kilo report;
- почему был создан этот V2 review;
- актуально ли это решение для всего проекта или только для конкретного направления;
- какой Boss Orchestrator управлял этой работой;
- какие блоки уже выполнены;
- какие решения были приняты внутри этой большой задачи;
- какие отчёты являются рабочими, а какие историческими;
- где находится текущий статус именно этой большой работы.

Подпроект решает эту проблему. Он становится смысловым контейнером, внутри которого живёт вся связанная работа.

---

## Что такое подпроект

**Подпроект** — это долговременная рабочая область внутри репозитория, созданная для одной крупной задачи, инициативы или направления работ.

Он отвечает на вопрос:

> “К чему относится вся эта цепочка Planning Chat → Task Control Pack → Boss Orchestrator → B1-блоки → младшие оркестраторы → Kilo/V1/V2/V3 → отчёты → решения?”

Подпроект — это не обязательно отдельный продуктовый модуль и не обязательно отдельная папка с кодом. Это именно **организационная и контекстная область работы**.

Он может быть связан с одной частью кода, с одним workflow, с одним направлением разработки или с одним крупным изменением.

Примеры подпроектов:

```text
Подпроект: Перестройка B1/Boss orchestration
Подпроект: Token diagnostics для экономии Codex-контекста
Подпроект: V3 helper scripts hardening
Подпроект: Table map editor architecture
Подпроект: Obsidian graph memory integration
Подпроект: Runtime move pipeline stabilization
```

---

## Место подпроекта в общей иерархии

После добавления подпроекта вся система выглядит так:

```text
Repository / основной проект
    ↓
Подпроект
    ↓
Planning Chat
    ↓
Task Control Pack
    ↓
Boss Orchestrator Chat
    ↓
B1 Block Chats
    ↓
Junior Orchestrators
    ↓
Kilo / V1 / V2 / V3 / локальные проверки
```

Подпроект находится между всем репозиторием и конкретной большой задачей выполнения.

Если раньше структура выглядела так:

```text
Repo
    ↓
Boss Orchestrator
    ↓
B1 Blocks
```

то теперь она становится точнее:

```text
Repo
    ↓
Подпроект
    ↓
Boss Orchestrator
    ↓
B1 Blocks
```

Это важно, потому что Boss Orchestrator не должен управлять всем репозиторием. Он должен управлять конкретным подпроектом или конкретной большой задачей внутри подпроекта.

---

## Отличие подпроекта от репозитория

Репозиторий — это технический контейнер всего проекта.

В нём находятся:

- исходный код;
- документация;
- `.ai`-память;
- scripts;
- contracts;
- prototypes;
- reports;
- workflow rules;
- разные направления разработки.

Подпроект — это логическая область работы внутри репозитория.

Он не обязан совпадать с папкой кода. Один подпроект может затрагивать несколько папок. И наоборот, одна папка кода может участвовать в разных подпроектах в разное время.

Репозиторий отвечает на вопрос:

> “Где живёт весь проект?”

Подпроект отвечает на вопрос:

> “Над какой крупной частью проекта мы сейчас работаем?”

---

## Отличие подпроекта от обычной задачи

Обычная задача может быть маленькой:

```text
исправить кнопку
добавить проверку
поправить текст
обновить один файл
```

Подпроект крупнее. Он содержит не одно действие, а управляемую цепочку действий.

Обычная задача часто может быть решена напрямую:

```text
Codex → Kilo → review → accept
```

Подпроект обычно требует:

```text
Planning Chat
Task Control Pack
Boss Orchestrator
B1-блоки
журнал
решения
отчёты
корректировки плана
```

Подпроект стоит создавать, если работа:

- многоэтапная;
- длится больше одного короткого запуска;
- требует нескольких чатов;
- требует нескольких B1-блоков;
- затрагивает важные решения;
- имеет внутренний журнал;
- может менять план по ходу выполнения;
- требует возвращения к ней позже;
- имеет собственную историю решений.

---

## Отличие подпроекта от B1-блока

Подпроект — это вся большая рабочая область.

B1-блок — это один ограниченный блок внутри этой области.

Пример:

```text
Подпроект:
  Перестройка B1/Boss orchestration

B1-блоки:
  B1-001 — описать новую концепцию процесса
  B1-002 — подготовить process docs
  B1-003 — внедрить token diagnostics
  B1-004 — провести пилотный тест
  B1-005 — обновить navigation и итоговые правила
```

Если B1-блок можно сравнить с “этапом работ”, то подпроект — это “папка всей инициативы”.

---

## Отличие подпроекта от Task Control Pack

Task Control Pack — это пакет документов, созданный после Planning Chat для запуска выполнения.

Подпроект — это более широкая сущность.

Внутри подпроекта может быть один Task Control Pack, а может быть несколько версий плана или несколько фаз. Task Control Pack — это один из ключевых артефактов подпроекта, но не весь подпроект.

Подпроект включает:

- текущий статус;
- план;
- Task Control Pack;
- Boss-журнал;
- B1-блоки;
- отчёты;
- решения;
- внешние материалы;
- финальный отчёт;
- архивные материалы.

---

## Отличие подпроекта от V1/V2/V3

V1, V2 и V3 — это инструменты или workflow-режимы.

Подпроект — это контейнер, внутри которого эти инструменты используются.

Например, в одном подпроекте может быть:

```text
V1 — для поиска архитектурного мнения
V2 — для ревью результата Kilo
V3 — для создания docs-only package
Kilo — для локальных изменений
B1 — для выполнения блока
Boss — для управления всей цепочкой
```

V1/V2/V3 не должны быть центром смысла. Они отвечают на вопрос “каким способом была выполнена часть работы”.

Подпроект отвечает на вопрос “к какой большой работе это относится”.

---

## Отличие подпроекта от Obsidian

Obsidian может быть человеческой картой смысла. Он может показывать связи между подпроектами, решениями, идеями и процессами.

Но подпроект внутри репозитория — это операционная рабочая область.

Разница такая:

```text
Подпроект в repo:
  операционная правда выполнения

Obsidian:
  человеческая карта понимания, связей и смыслов
```

Obsidian может ссылаться на подпроект, объяснять его место в общей картине и связывать его с другими темами. Но Obsidian не должен заменять рабочие журналы, block reports, Task Control Pack и фактические документы выполнения.

---

## Когда создавать подпроект

Подпроект не нужен для каждой мелкой задачи.

Подпроект стоит создавать, если выполняется хотя бы несколько условий:

- задача большая;
- есть несколько этапов;
- план может изменяться;
- нужен Boss Orchestrator;
- нужны B1-блоки;
- есть несколько V1/V2/V3/Kilo цепочек;
- работа длится больше одного дня;
- нужно сохранить историю решений;
- есть риск потерять контекст;
- работа относится к отдельному направлению внутри большого проекта;
- нужно будет вернуться к этой теме позже;
- требуется отдельный журнал;
- есть несколько чатов, относящихся к одной теме.

Подпроект не стоит создавать, если задача маленькая и локальная:

```text
поправить опечатку
изменить одну строку
обновить небольшой текст
сделать простой одноразовый Kilo-handoff
```

## Жизненный цикл подпроекта

Подпроект может проходить несколько стадий.

### 1. Идея

Есть крупная тема, но она ещё не оформлена.

На этом этапе можно обсуждать, стоит ли создавать подпроект.

### 2. Создание подпроекта

Фиксируется, что это отдельная рабочая область.

Определяются:

- название;
- цель;
- границы;
- владелец управления;
- начальный статус;
- где будут храниться документы;
- какие материалы уже существуют.

### 3. Планирование

Planning Chat исследует задачу, вызывает V1 при необходимости, формирует полное понимание и готовит Task Control Pack.

### 4. Передача Boss Orchestrator

Planning Chat завершает свою роль. Создаётся новый Boss Orchestrator chat, который получает подготовленный контекст подпроекта.

### 5. Выполнение блоков

Boss запускает B1-блоки, принимает отчёты, обновляет short plan, ведёт журнал и decision log.

### 6. Корректировка

Если по ходу выполнения появляются новые факты, Boss меняет план подпроекта явно и фиксирует решения.

### 7. Финализация

Когда все блоки выполнены, Boss готовит final report.

### 8. Архивация

Подпроект переводится в завершённое или архивное состояние.

Полные материалы остаются доступны, но не являются активным рабочим контекстом.

---

## Возможные статусы подпроекта

У подпроекта должен быть простой статус.

Рекомендуемые статусы:

```text
idea
planned
active
blocked
paused
completed
cancelled
archived
```

Также полезно отдельно указывать фазу:

```text
planning
task-control-pack-ready
boss-orchestration
block-execution
review
finalization
archived
```

Пример:

```text
Status: active
Phase: block-execution
Current Boss: Boss Orchestrator Chat for TS-...
Current Block: B1-002
```

---

## Что должно быть главным документом подпроекта

У подпроекта должен быть короткий главный документ состояния.

Его можно мыслить как “паспорт подпроекта”.

Он должен отвечать на вопросы:

- что это за подпроект;
- зачем он существует;
- какая цель;
- что входит в scope;
- что не входит в scope;
- какой текущий статус;
- какая текущая фаза;
- где полный план;
- где короткий план;
- где index блоков;
- где journal;
- где decision log;
- какой следующий шаг;
- какие главные риски;
- какие чаты или блоки активны;
- где лежат полные источники.

Этот документ должен быть коротким. Его можно читать часто.

Он не должен превращаться в полный отчёт обо всём.

---

## Рабочие документы подпроекта

Внутри подпроекта могут жить следующие документы.

### 1. Главный статус подпроекта

Короткий файл текущего состояния.

### 2. Подробный план

Полное описание выбранной стратегии.

### 3. Короткий рабочий план

Актуальная сжатая версия плана, которую Boss может обновлять.

### 4. Индекс блоков

Список B1-блоков, их статусы, зависимости и ссылки на handoff/report.

### 5. Журнал сессии

Хронология управления подпроектом.

### 6. Decision log

Список принятых решений.

### 7. Boss bootstrap

Документ для запуска нового Boss Orchestrator chat.

### 8. Handoff для B1-блоков

Отдельные документы для каждого B1-блока.

### 9. Block reports

Отчёты B1-блоков.

### 10. Внешние материалы

Ссылки или копии V1/V2/V3 материалов, если они относятся к подпроекту.

### 11. Kilo reports

Отчёты Kilo-задач, относящиеся к подпроекту.

### 12. Final report

Итоговый отчёт подпроекта.

## Как подпроект помогает экономить Codex-контекст

Подпроект является контекстной границей.

Вместо команды:

```text
прочитай весь репозиторий и всю .ai
```

можно сказать:

```text
работай в рамках подпроекта X
сначала прочитай только главный статус подпроекта,
короткий план,
индекс блоков
и handoff текущего блока
```

Это резко сокращает рабочий контекст.

Подпроект позволяет Codex не искать по всему репозиторию, к чему относится задача. Он сразу видит ограниченную область:

- какие документы актуальны;
- какие блоки связаны;
- какие решения действуют;
- какие отчёты относятся к этой работе;
- что не относится к этой работе.

Подпроект также защищает от смешивания старых и новых материалов. Если V2-ответ относится к другому подпроекту, его не надо читать для текущего подпроекта.

---

## Как подпроект помогает качеству

Подпроект не только экономит токены. Он улучшает качество управления.

Он даёт:

- единое место для текущего статуса;
- явную историю решений;
- понятную карту блоков;
- связь между B1-блоками и общей целью;
- возможность понять, что уже сделано;
- возможность восстановить контекст после перерыва;
- защиту от случайного смешивания задач;
- понятное место для финального отчёта.

Без подпроекта Boss Orchestrator может управлять блоками, но не иметь надёжной “папки смысла”. С подпроектом каждый блок, отчёт и решение получают место.

---

## Подпроект как дом для чатов

Один подпроект может включать много чатов:

```text
Planning Chat
Boss Orchestrator Chat
B1 Block Chat 1
B1 Block Chat 2
B1 Block Chat 3
Junior Orchestrator Chat / runs
External V1 chats
External V2 review chats
V3 artifact-producing chats
Kilo runs
```

Но все эти чаты относятся к одному подпроекту.

Поэтому каждый отчёт или документ должен указывать, к какому подпроекту он относится.

Это помогает избежать ситуации, когда через неделю невозможно понять, почему был создан тот или иной файл.

---

## Подпроект и границы scope

У подпроекта обязательно должны быть границы.

Нужно фиксировать:

- что входит в подпроект;
- что явно не входит;
- какие папки/файлы могут затрагиваться;
- какие темы не должны решаться внутри этого подпроекта;
- какие решения относятся только к подпроекту, а какие влияют на весь репозиторий.

Это особенно важно, потому что подпроект может быть крупным и соблазнять агентов расширять работу.

Если подпроект называется “Token diagnostics”, он не должен случайно превратиться в “переписать весь AI workflow”.

Если подпроект называется “V3 helper scripts hardening”, он не должен случайно создать `apply_v3_package.py`, если это запрещено.

---

## Подпроект и право менять план

Boss Orchestrator внутри подпроекта может менять план, если появились новые факты.

Но он должен отличать два типа изменений:

### Локальное изменение внутри подпроекта

Например:

```text
добавить repair-блок
поменять порядок B1-блоков
уточнить критерий приёмки
```

Такое изменение Boss может принять сам, если оно не выходит за scope и не требует решения человека.

### Изменение, влияющее на весь проект

Например:

```text
изменить общий V3 contract
изменить правила Kilo/Codex
ввести новый обязательный workflow
перенести архитектурную границу
```

Такое изменение должно быть явно поднято выше, потому что оно может затронуть другие подпроекты или весь repo workflow.

Подпроект не должен самовольно менять глобальные правила, если это не было его целью.

---

## Подпроект и архив

После завершения подпроекта его не нужно удалять.

Он становится архивом:

- что планировали;
- что сделали;
- какие решения приняли;
- какие блоки были;
- какие проблемы возникали;
- какие выводы полезны для будущего.

Но архивный подпроект не должен быть активным рабочим контекстом по умолчанию.

Новый подпроект может ссылаться на старый, но не должен читать его полностью без необходимости.

---

## Где подпроект может жить в репозитории

Конкретная структура может быть выбрана позже. По смыслу нужна отдельная зона внутри `.ai`.

Возможные варианты:

```text
.ai/subprojects/
```

или:

```text
.ai/taskspaces/
```

или:

```text
.ai/workstreams/
```

Так как выбран термин “подпроект”, естественный вариант:

```text
.ai/subprojects/
```

Пример возможной структуры:

```text
.ai/subprojects/
  SP-20260529-b1-boss-orchestration/
    SUBPROJECT_STATE.md
    FULL_PLAN.md
    SHORT_PLAN.md
    BLOCKS_INDEX.md
    DECISION_LOG.md
    SESSION_JOURNAL.md
    BOSS_BOOTSTRAP.md
    blocks/
      B1-001/
        HANDOFF.md
        BLOCK_REPORT.md
      B1-002/
        HANDOFF.md
        BLOCK_REPORT.md
    external/
      v1/
      v2/
      v3/
    kilo/
      reports/
    final/
      FINAL_REPORT.md
```

Эта структура не является обязательным шаблоном. Она показывает идею: у каждой крупной работы есть свой дом.

---

## ID подпроекта

Полезно иметь отдельный ID подпроекта.

Например:

```text
SP-20260529-b1-boss-orchestration
SP-20260529-token-diagnostics
SP-20260529-v3-helper-scripts-hardening
```

Префикс `SP` можно понимать как `SubProject`.

ID нужен, чтобы связывать между собой:

- Planning Chat;
- Boss Chat;
- B1-блоки;
- V1/V2/V3;
- Kilo reports;
- решения;
- финальный отчёт.

Если у каждого материала есть `Subproject ID`, легче понять, куда он относится.

---

## Что должен видеть новый чат при входе в подпроект

Когда создаётся новый Boss Orchestrator или B1-чат, ему не нужно давать всю историю подпроекта.

Ему нужно дать входной контекст уровня его роли.

### Для Boss Orchestrator

Нужны:

- ID подпроекта;
- статус подпроекта;
- short plan;
- block index;
- session journal summary;
- decision log;
- текущий следующий блок;
- правила чтения full materials только при необходимости.

### Для B1-блока

Нужны:

- ID подпроекта;
- ID блока;
- цель блока;
- границы блока;
- входные материалы блока;
- критерии приёмки;
- что вернуть Boss-у;
- ссылки на full plan только если нужны.

### Для младшего оркестратора

Нужны:

- ID подпроекта;
- ID блока;
- ID подзадачи;
- конкретная подзадача;
- ограничения;
- ожидаемый отчёт;
- какие инструменты разрешены.

---

## Подпроект и завершение задачи

Когда большая работа завершена, подпроект должен иметь понятное финальное состояние.

Финализация подпроекта должна отвечать:

- достигнута ли цель;
- какие блоки выполнены;
- какие блоки отменены;
- какие решения приняты;
- какие артефакты созданы;
- какие файлы или области затронуты;
- какие проверки выполнены;
- какие риски остались;
- что требуется от человека;
- есть ли follow-up подпроекты;
- можно ли архивировать подпроект.

Подпроект не должен завершаться словами “вроде всё сделано”. Он должен завершаться финальным отчётом.

---

## Как подпроект меняет всю систему

После введения подпроекта вся система становится не просто цепочкой чатов, а системой рабочих областей.

Было:

```text
Planning Chat → Boss → B1 → Kilo/V1/V2/V3
```

Становится:

```text
Repository
    ↓
Подпроект
    ↓
Planning Chat
    ↓
Task Control Pack
    ↓
Boss Orchestrator
    ↓
B1 Blocks
    ↓
Junior Orchestrators
    ↓
Kilo / V1 / V2 / V3
```

Это делает систему более устойчивой.

Теперь каждый элемент имеет адрес:

```text
этот V1 относится к подпроекту SP-...
этот B1 относится к подпроекту SP-...
этот Kilo report закрывает B1-002 внутри SP-...
это решение принято в рамках SP-...
этот final report завершает SP-...
```

---

## Главная мысль про подпроект

Подпроект — это “дом” большой задачи.

Он не заменяет Boss Orchestrator, B1 или Task Control Pack. Он даёт им общую область принадлежности.

Без подпроекта система может работать, но постепенно начинает теряться связь между артефактами.

С подпроектом появляется ясность:

```text
где живёт эта работа;
что к ней относится;
какой у неё статус;
кто ей управляет;
какие блоки есть;
какие решения приняты;
где искать отчёты;
когда она завершена.
```

Если Boss Orchestrator — это управляющий, а B1 — это исполнительный штаб блока, то подпроект — это кабинет, папка и карта всей большой работы.

## Назначение документа

Этот документ описывает общую идею многоуровневой системы AI-разработки, в которой большая задача не выполняется одним бесконечным чатом, а проходит через несколько управляемых уровней:

1. планировочный чат;
2. пакет управления задачей;
3. новый Boss Orchestrator chat;
4. отдельные B1 block chats;
5. младшие оркестраторы;
6. инструменты исполнения и ревью: Kilo, V1, V2, V3 и другие режимы;
7. журналы, отчёты, решения и контролируемые обновления плана.

Документ не содержит конкретных промптов, шаблонов или copy-paste инструкций. Его задача — объяснить саму систему: зачем она нужна, как она должна работать, какие роли в ней существуют, где проходят границы ответственности, как экономится контекст Codex, как сохраняется контроль качества и почему эта система не должна превращаться в хаос или бюрократию.

Этот документ должен быть понятен человеку или агенту, который впервые видит систему и должен быстро понять её философию, архитектуру и правила поведения.

---

## Короткая суть системы

Система строится вокруг одной идеи:

> Большая задача не должна жить целиком в одном длинном Codex-чате.

Вместо этого большая задача сначала тщательно продумывается в отдельном планировочном чате. Этот чат может быть дорогим по токенам, медленным, исследовательским и неэкономным. Его задача — не выполнить работу, а создать хороший план.

Когда план готов, планировочный чат завершает свою роль. Он не тянется до конца задачи. Вместо этого он создаёт набор управленческих документов: подробный план, короткий рабочий план, список блоков, handoff для каждого блока, журнал и специальный документ передачи контекста в новый чат.

После этого создаётся новый чат — Boss Orchestrator. Он стартует с чистого контекста и получает не всю историю обсуждения, а только подготовленный пакет управления задачей. Boss Orchestrator ведёт выполнение задачи по блокам, принимает отчёты, обновляет журнал решений, корректирует план при необходимости и запускает следующие B1-блоки.

Каждый B1-блок выполняется в отдельном новом чате. B1-чат получает только свой блок, сам планирует выполнение блока, делит его на 2–4 подзадачи, нанимает младших оркестраторов, принимает их отчёты, проверяет результаты и в конце возвращает Boss Orchestrator-у отчёт блока.

Младшие оркестраторы решают отдельные подзадачи. Они могут использовать Kilo, V1, V2, V3 или собственный анализ, но не должны тянуть на себя весь блок или всю большую задачу.

Вся система работает как иерархия:

```text
Planning Chat
    ↓
Task Control Pack
    ↓
Boss Orchestrator Chat
    ↓
B1 Block Chat 1
    ↓
Junior Orchestrators
    ↓
Kilo / V1 / V2 / V3 / локальная проверка
```

Главная цель — не просто “расписать работу красиво”. Главная цель — управлять контекстом, снизить нагрузку на Codex, сохранить качество проверки и не потерять стратегический смысл большой задачи.

---

## Почему такая система вообще нужна

AI-разработка быстро сталкивается с несколькими проблемами.

### 1. Длинный контекст начинает вредить

Если всё делать в одном чате, он постепенно заполняется:

- первоначальными идеями;
- ошибочными предположениями;
- длинными обсуждениями;
- альтернативными вариантами;
- ответами внешних чатов;
- отчётами Kilo;
- ревью;
- исправлениями;
- техническими деталями;
- спорами о процессе;
- промежуточными планами;
- устаревшими решениями.

В какой-то момент чат перестаёт быть чистым рабочим инструментом. Он превращается в архив всего, что когда-либо обсуждалось. Агенту приходится каждый раз тащить за собой слишком много текста, часть которого уже неактуальна.

Даже если модель технически способна удерживать большой контекст, это не значит, что это эффективно. Чем больше лишнего контекста, тем выше риск:

- спутать старое решение с новым;
- принять промежуточную идею за финальную;
- упустить важное ограничение;
- начать чинить не ту проблему;
- прочитать слишком много и потратить лишние токены;
- начать “улучшать” то, что трогать не надо;
- забыть, где заканчивается план и начинается выполнение.

### 2. Codex дорог как главный контекстный ресурс

В этой системе важно не обязательно экономить все токены всех инструментов. Можно сознательно тратить токены внешних чатов, Kilo или вспомогательных агентов, если это помогает разгрузить главный Codex-контекст.

Главный дорогой ресурс — это контекст того агента, который принимает стратегические решения, проверяет качество, связывает блоки между собой и отвечает за итоговую целостность задачи.

Если Codex постоянно читает всё: полные V1-ответы, полные V2-ревью, все Kilo-отчёты, все обсуждения и все старые планы, экономии нет. Если же Codex получает только управленческие сводки, изменённые файлы, решения и необходимые проверки, то он тратит свои токены на действительно важное.

### 3. Kilo и другие исполнители могут ошибаться

Система не строится на доверии к более слабым исполнителям. Kilo может ошибаться. Внешний чат может дать сильное ревью, но не иметь прямого доступа к локальному состоянию репозитория. Младший оркестратор может неправильно понять задачу. Поэтому нужна не только делегация, но и проверка.

Смысл системы не в том, чтобы сказать: “Kilo сделал, значит готово”. Смысл в другом:

- Kilo выполняет локальную работу;
- младший оркестратор проверяет первично;
- B1-оркестратор проверяет результат блока;
- Boss Orchestrator принимает блок на уровне всей задачи;
- человек остаётся финальным владельцем решений.

### 4. Большие задачи требуют не просто исполнения, а управления

Маленькую задачу можно выполнить напрямую: дать Kilo короткий handoff, получить diff, проверить, принять.

Большая задача обычно требует:

- исследования;
- архитектурного решения;
- разбиения на блоки;
- проверки рисков;
- последовательности;
- понимания зависимостей;
- журналирования;
- контроля scope;
- ревью промежуточных результатов;
- принятия решений по ходу выполнения.

Именно для таких задач нужна иерархическая система.

## Главная философия системы

Система строится на нескольких принципах.

### Принцип 1. Один чат не должен быть всем

Один чат не должен одновременно быть:

- местом исследования;
- местом планирования;
- местом выполнения;
- местом проверки;
- архивом внешних ответов;
- журналом решений;
- рабочей памятью всех блоков;
- финальным reviewer-ом.

Когда всё смешивается в одном месте, появляется шум. Поэтому система разделяет роли.

### Принцип 2. После планирования нужен контекстный разрыв

Планировочный чат должен завершиться после того, как подготовлен качественный пакет управления задачей.

Он не должен тянуться дальше в выполнение, потому что в нём уже накопилось много чернового и исследовательского контекста.

Правильная логика:

```text
Планировочный чат думает дорого и подробно.
Затем он создаёт Task Control Pack.
Затем он завершает свою роль.
Дальше работает новый Boss Orchestrator chat.
```

### Принцип 3. Дальше передаётся не история, а подготовленный пакет

Новый Boss Orchestrator не получает всю переписку планировочного чата. Он получает только подготовленный пакет управления задачей:

- полное описание плана;
- короткую рабочую версию плана;
- список блоков;
- границы;
- решения;
- handoff для блоков;
- журнал;
- инструкции по управлению выполнением;
- указания, где лежат полные источники при необходимости.

История планирования остаётся архивной. Она не является обязательным рабочим контекстом.

### Принцип 4. Каждый уровень знает только свой уровень

Boss Orchestrator не должен тащить весь мусор B1-блоков.  
B1-блок не должен тащить всю историю planning chat.  
Младший оркестратор не должен тащить весь блок.  
Kilo не должен получать всю стратегическую задачу, если ему нужно изменить один файл.

Каждый уровень получает ровно тот контекст, который нужен ему для его роли.

### Принцип 5. Полные материалы сохраняются, но не читаются по умолчанию

Полные V1-ответы, V2-ревью, Kilo-отчёты, логи и внешние ответы должны сохраняться в `.md` файлах. Это важно для аудита, восстановления и проверки.

Но они не должны автоматически возвращаться в главный Codex-контекст. По умолчанию наверх должны идти:

- короткие отчёты;
- управленческие выводы;
- список изменений;
- риски;
- решения;
- ссылки на полные источники.

Полный источник открывается только если есть причина.

### Принцип 6. Делегация не отменяет проверку

Краткий отчёт или “капсульная верхушка” не является доказательством. Это навигация для проверки.

Если Kilo пишет, что всё сделал, это не значит, что всё действительно сделано. B1-оркестратор или Boss Orchestrator должен сверить:

- изменённые файлы;
- scope;
- заявленные критерии приёмки;
- тесты;
- возможные побочные эффекты;
- соответствие плану;
- остаточные риски.

### Принцип 7. План можно менять, но только осознанно

Большой план не должен быть тюрьмой. По ходу выполнения могут обнаружиться новые факты, ошибки, ограничения, зависимости или лучшие решения.

Boss Orchestrator должен иметь право корректировать план. Но корректировка должна быть явной:

- что изменилось;
- почему;
- какие блоки затронуты;
- какие риски;
- кто принимает решение;
- как обновляется short plan или decision log.

План не должен “тихо поплыть”.

## Основные роли системы

## 1. Planning Chat

Planning Chat — это первый большой чат, где задача осмысляется.

Это самый свободный и самый дорогой по токенам этап. Здесь можно:

- долго обсуждать задачу;
- вызывать V1 много раз;
- проверять разные подходы;
- искать противоречия;
- строить архитектуру;
- сравнивать варианты;
- задавать стратегические вопросы;
- уточнять, что вообще должно быть сделано.

Planning Chat не обязан быть экономным. Его задача — качество мысли.

Но он должен понимать свою конечную цель: не “решить всё до конца в чате”, а подготовить управленческий пакет для следующего этапа.

### Что делает Planning Chat

Planning Chat:

1. принимает большую задачу;
2. уточняет её смысл;
3. собирает ограничения;
4. при необходимости вызывает V1 для внешних мнений;
5. формирует архитектурное понимание;
6. определяет крупные блоки работы;
7. фиксирует принятые решения;
8. выявляет риски;
9. готовит подробный план;
10. готовит короткий рабочий план;
11. готовит список B1-блоков;
12. готовит handoff для каждого блока;
13. готовит bootstrap для Boss Orchestrator;
14. создаёт начальный журнал/decision log;
15. завершает свою роль.

### Что Planning Chat не должен делать

Planning Chat не должен:

- вести всю реализацию до конца;
- становиться вечным управляющим чатом;
- принимать отчёты от всех B1-блоков;
- читать все будущие логи;
- быть местом бесконечного накопления контекста;
- смешивать черновую историю с рабочими документами выполнения.

### Почему Planning Chat должен завершаться

Planning Chat по природе содержит много “грязного” контекста:

- черновые идеи;
- неверные предположения;
- альтернативы, которые потом отклонены;
- внешние ответы;
- промежуточные выводы;
- рассуждения, которые были нужны только для поиска решения.

Эти материалы полезны для мышления, но вредны как постоянный рабочий контекст. Поэтому после готовности плана они должны быть сжаты в Task Control Pack, а сам чат должен быть закрыт как активный управляющий контекст.

---

## 2. Task Control Pack

Task Control Pack — это набор документов, который Planning Chat передаёт новому Boss Orchestrator chat.

Это не обязательно должен быть один файл. Это может быть комплект `.md` документов. Важна не конкретная структура, а назначение: пакет должен позволить новому чату продолжить задачу без чтения всей истории planning chat.

### Зачем нужен Task Control Pack

Он нужен, чтобы:

- отделить итог планирования от процесса планирования;
- передать только актуальную версию понимания;
- обеспечить чистый старт Boss Orchestrator;
- сохранить подробный план как справочник;
- дать короткий рабочий план для ежедневного управления;
- подготовить блоки к запуску;
- сохранить решения и ограничения;
- дать путь к корректировке плана.

### Что должен содержать Task Control Pack

Конкретные имена файлов могут быть любыми, но по смыслу пакет должен включать несколько типов материалов.

#### 1. Подробный план

Полный, развёрнутый план большой задачи.

Он нужен как справочник. В нём могут быть:

- подробное описание цели;
- контекст;
- выбранная стратегия;
- архитектурные решения;
- альтернативы и почему они отвергнуты;
- блоки;
- зависимости;
- риски;
- критерии готовности;
- ожидаемый итог.

Подробный план не обязательно должен читаться на каждом шаге. Он существует как источник глубокой информации.

#### 2. Короткий рабочий план

Это главный документ для Boss Orchestrator.

Он должен быть намного короче полного плана. В нём должны быть:

- цель задачи;
- текущий статус;
- список блоков;
- порядок выполнения;
- ключевые решения;
- текущие ограничения;
- открытые вопросы;
- следующий блок;
- последние принятые изменения плана.

Короткий план должен быть живым документом. Boss Orchestrator может обновлять его по ходу работы.

#### 3. Индекс блоков

Список B1-блоков с их назначением и статусом.

Он нужен, чтобы Boss Orchestrator видел:

- какие блоки запланированы;
- какие уже выполнены;
- какие заблокированы;
- какие требуют пересмотра;
- какой handoff соответствует каждому блоку;
- какие зависимости есть между блоками.

#### 4. Handoff для каждого B1-блока

Каждый блок должен иметь свой стартовый контекст.

B1-handoff должен быть самодостаточным: новый B1-чат должен понять свой блок без чтения всего planning chat.

В handoff должны быть:

- цель блока;
- границы блока;
- входные данные;
- что можно менять;
- что нельзя менять;
- ожидаемый результат;
- критерии приёмки;
- ограничения;
- что вернуть Boss Orchestrator-у.

#### 5. Bootstrap для Boss Orchestrator

Это документ передачи контекста в новый Boss-чат.

Он должен объяснить новому чату его роль:

- он не является planning chat;
- он не должен перечитывать всю историю;
- он управляет выполнением задачи по Task Control Pack;
- он ведёт журнал;
- он запускает B1-блоки;
- он принимает отчёты блоков;
- он может корректировать план при необходимости;
- он должен сохранять решения;
- он должен не втягиваться в операционный шум.

#### 6. Журнал сессии

Начальный журнал, который дальше ведёт Boss Orchestrator.

Он должен фиксировать:

- старт задачи;
- старт блоков;
- отчёты блоков;
- решения;
- изменения плана;
- проблемы;
- принятые результаты;
- финальное состояние.

#### 7. Decision log

Журнал принятых решений.

Он может быть отдельным документом или частью короткого рабочего плана.

Его задача — сохранять не все рассуждения, а именно решения:

- что принято;
- что отклонено;
- почему;
- когда;
- чем это влияет на следующие блоки.

---

## 3. Boss Orchestrator Chat

Boss Orchestrator Chat — это новый чат, который создаётся после завершения Planning Chat.

Он получает Task Control Pack и управляет выполнением всей большой задачи.

Это уже не исследовательский чат. Это управляющий чат.

### Основная роль Boss Orchestrator

Boss Orchestrator:

- читает bootstrap;
- принимает короткий план;
- понимает список блоков;
- запускает B1-блоки;
- принимает block reports;
- проверяет результаты блоков;
- обновляет журнал;
- обновляет короткий план;
- фиксирует решения;
- корректирует план при необходимости;
- решает, когда запускать следующий блок;
- завершает задачу, когда все блоки выполнены или когда задача заблокирована.

### Что Boss Orchestrator должен читать по умолчанию

Boss Orchestrator не должен читать всё подряд.

По умолчанию ему нужны:

- bootstrap;
- короткий рабочий план;
- индекс блоков;
- handoff текущего блока;
- отчёт текущего блока;
- список решений;
- при необходимости — конкретный фрагмент подробного плана.

Полный planning chat не нужен.

Полные V1/V2 ответы не нужны по умолчанию.

Полные Kilo-логи не нужны по умолчанию.

### Что Boss Orchestrator может читать при необходимости

Boss может углубиться, если:

- отчёт блока противоречив;
- B1 не справился;
- есть подозрение на ошибку;
- план устарел;
- требуется важное решение;
- нужно проверить исходное обоснование;
- human попросил глубже разобраться;
- изменён риск или scope.

Тогда Boss может открыть:

- подробный план;
- полный отчёт B1;
- внешнее ревью;
- Kilo report;
- diff;
- контракт;
- другой исходный документ.

Но это должно быть осознанное углубление, а не автоматическое чтение всего архива.

### Boss Orchestrator как владелец живого плана

Boss не просто запускает блоки по очереди. Он должен следить, остаётся ли план правильным.

Если блок показал, что исходное решение ошибочно, Boss должен:

1. зафиксировать, что выяснилось;
2. определить, какие части плана затронуты;
3. предложить корректировку;
4. понять, нужно ли решение человека;
5. обновить короткий план;
6. обновить индекс блоков;
7. продолжить выполнение уже по новой версии плана.

Boss — не пассивный диспетчер. Он владелец актуального управленческого состояния задачи.

### Что Boss Orchestrator не должен делать

Boss не должен:

- сам выполнять весь кодовый блок, если для этого нужен B1/Kilo;
- читать все сырые логи без причины;
- переписывать историю решений задним числом;
- запускать слишком много параллельных блоков без контроля;
- менять план молча;
- принимать отчёты Kilo без проверки через B1;
- превращаться в новый бесконечный planning chat;
- терять связь с исходной целью задачи.

---

## 4. B1 Block Chat

B1 Block Chat — это отдельный чат для одного конкретного блока.

Он создаётся Boss Orchestrator-ом или человеком по handoff-у блока. Он не должен получать весь Task Control Pack целиком, если это не нужно. Он получает только свой block handoff и, возможно, минимальные ссылки на общие документы.

### Главная роль B1

B1-чат — это оперативный штаб блока.

Он должен:

1. понять цель блока;
2. проверить границы блока;
3. построить план выполнения блока;
4. определить 2–4 подзадачи;
5. под каждую подзадачу нанять младшего оркестратора;
6. определить, какие инструменты нужны: Kilo, V1, V2, V3 или собственный анализ;
7. принять отчёты младших оркестраторов;
8. проверить результаты;
9. запросить repair, если нужно;
10. завершить блок;
11. подготовить block report для Boss Orchestrator.

### Почему B1 живёт в отдельном чате

Отдельный B1-чат нужен для изоляции контекста.

Блок может быть сложным. Внутри него могут быть:

- несколько подзадач;
- Kilo-задачи;
- внешние V1/V2 запросы;
- ошибки;
- исправления;
- промежуточные отчёты;
- технические детали;
- локальные проверки.

Если всё это тащить в Boss Orchestrator, Boss быстро загрязнится. Поэтому B1-чат живёт отдельно, работает внутри блока, а потом возвращает наверх только управленческий отчёт.

### Ограничение B1

Один B1-чат должен отвечать за один блок.

Когда блок завершён, B1-чат должен завершить свою роль. Его не надо использовать для следующего блока, иначе контекст опять начнёт разрастаться.

Правило:

```text
Один B1-чат = один блок.
Блок завершён = отчёт наверх = чат закрыт.
```

### B1 как контролёр младших оркестраторов

B1 не просто пересылает подзадачи. Он отвечает за качество блока.

Он должен:

- проверять, что младший оркестратор понял задачу;
- проверять, что Kilo не вышел за scope;
- проверять, что результат соответствует критериям;
- видеть, какие риски остались;
- решать, нужна ли доработка;
- не отдавать наверх сырой непроверенный результат.

B1 — это первый серьёзный фильтр качества после исполнителей.

---

## 5. Младший оркестратор

Младший оркестратор — это агент, которому B1 поручает одну конкретную подзадачу.

Он не должен владеть всем блоком. Он не должен принимать стратегические решения всей задачи. Он должен выполнить ограниченный кусок работы и вернуть B1 понятный отчёт.

### Что делает младший оркестратор

В зависимости от подзадачи он может:

- подготовить Kilo-handoff;
- запросить V1-мнение;
- подготовить V2-review request;
- использовать V3 artifact flow;
- проанализировать отдельный технический вопрос;
- проверить результат Kilo;
- собрать отчёт;
- предложить доработку.

### Почему младший оркестратор — именно оркестратор, а не просто исполнитель

Потому что даже маленькая подзадача может требовать нескольких действий.

Например:

1. понять проблему;
2. сформулировать Kilo-задачу;
3. принять Kilo report;
4. проверить его;
5. при необходимости запросить repair;
6. вернуть B1 краткий, но достаточный отчёт.

Это не просто “напиши код”. Это маленький цикл управления.

### Ограничения младшего оркестратора

Он должен:

- иметь одну чёткую подзадачу;
- не расширять scope;
- не тянуть весь блок;
- не принимать решения за Boss;
- не возвращать наверх сырые огромные материалы;
- сохранять полные ответы в файлы, если нужно;
- возвращать B1 сжатый отчёт с путями к источникам.

---

## 6. Kilo

Kilo — локальный исполнитель.

Его роль — выполнять конкретные задачи в репозитории:

- читать файлы;
- редактировать файлы;
- запускать команды;
- проверять результат;
- возвращать отчёт.

Kilo не является финальным судьёй качества. Его отчёт — это не доказательство, а материал для проверки.

### Как Kilo используется в системе

Обычно Kilo получает задачу от младшего оркестратора или B1. Задача должна быть ограниченной:

- какие файлы можно трогать;
- какие нельзя;
- что нужно сделать;
- какие проверки запустить;
- какой отчёт вернуть;
- какие риски указать.

### Почему нельзя слепо доверять Kilo

Потому что Kilo может:

- неправильно понять задачу;
- изменить лишние файлы;
- не запустить нужные тесты;
- неверно интерпретировать результат;
- не заметить edge case;
- сделать частичное исправление и назвать его полным.

Поэтому Kilo должен возвращать не просто “готово”, а материал для проверки:

- список изменённых файлов;
- что изменено;
- какие команды запускались;
- что не запускалось и почему;
- какие проблемы были;
- какие риски остались;
- какие места нужно проверить Codex/B1/Boss.

### Как Kilo помогает экономить Codex

Kilo может читать большие файлы, запускать локальные проверки и сохранять длинные отчёты в `.md`, не вставляя их в главный Codex-чат.

Экономия появляется, если наверх возвращается не полный лог, а короткий проверочный пакет:

- изменённые файлы;
- summary;
- статус критериев;
- тесты;
- риски;
- пути к полным источникам.

---

## 7. V1

V1 — внешний prompt-only режим для планирования, критики, идей, second opinion.

В этой системе V1 особенно полезен на этапе Planning Chat, потому что там задача — получить максимально качественный план, а не экономить токены.

### Где V1 уместен

V1 уместен:

- при выборе архитектуры;
- при поиске рисков;
- при сравнении подходов;
- при уточнении UX;
- при формулировании плана;
- при запросе независимой критики;
- при сложных вопросах, где полезно внешнее мнение.

### Где V1 не должен использоваться

V1 не должен заменять:

- локальную проверку файлов;
- review реального diff;
- Kilo execution;
- V2 bounded review snapshot;
- решение Boss Orchestrator.

V1 может помочь думать, но не подтверждает факт выполнения.

---

## 8. V2

V2 — внешний bounded review реального WIP snapshot.

Он нужен, когда уже есть конкретный результат, файл, diff или snapshot, и требуется внешнее senior review.

### Где V2 уместен

V2 уместен:

- после Kilo-реализации;
- перед принятием сложного блока;
- при проверке safety boundaries;
- при проверке contract compliance;
- при ревью архитектурных изменений;
- при сомнении в результате младшего оркестратора или Kilo.

### Важное ограничение V2

Внешний V2-чат может быть сильным, но он не является владельцем репозитория. Он не редактирует локальный workspace и может не иметь полной картины фактического состояния.

Поэтому V2-ответ — это input для проверки, а не автоматическое принятие.

B1 или Boss должен:

- прочитать выводы V2;
- сверить с фактическими файлами;
- принять, отклонить или частично принять замечания;
- при необходимости запустить repair.

---

## 9. V3

V3 — artifact-producing workflow.

В этой системе V3 может использоваться не только для product-code artifacts, но и для docs-only task control pack.

Это особенно полезно после Planning Chat.

### Почему V3 подходит для передачи плана

Planning Chat может создать V3 docs-only package, в котором будут:

- подробный план;
- короткий план;
- block index;
- boss bootstrap;
- handoff для блоков;
- journal/decision-log заготовки;
- проверочный prompt или post-import check, если нужен.

Это превращает результат планирования в структурированный набор файлов, а не в длинный кусок переписки.

### Важное ограничение

V3 не должен автоматически становиться реальным изменением репозитория без проверки. Если используется ZIP/artifact package, он должен проходить безопасный import/staging/review process.

## Жизненный цикл большой задачи

Ниже описан общий путь задачи через систему.

## Этап 1. Запуск большой задачи

Человек формулирует крупную задачу.

Planning Chat должен определить:

- задача действительно большая или её можно выполнить напрямую;
- нужен ли полноценный B1/Boss workflow;
- какие области затронуты;
- есть ли необходимость в V1-исследовании;
- нужно ли планировать несколько блоков.

Если задача маленькая, система не должна навязывать полный процесс. Полный B1/Boss workflow оправдан именно для крупных, рискованных или многоэтапных задач.

## Этап 2. Исследование и планирование

Planning Chat свободно работает над пониманием задачи.

На этом этапе можно не экономить токены. Важно получить качественный план.

Planning Chat может:

- собирать идеи;
- вызывать V1 много раз;
- анализировать ответы;
- спорить с ними;
- проверять альтернативы;
- уточнять общую архитектуру;
- думать о рисках;
- выделять блоки.

Но к концу этапа всё должно быть сведено в структурированный Task Control Pack.

## Этап 3. Создание Task Control Pack

Когда план готов, Planning Chat создаёт управленческий пакет.

Главная задача этого этапа — отделить итог планирования от процесса планирования.

Task Control Pack должен быть самодостаточным для нового Boss Orchestrator chat.

В этот момент Planning Chat должен считать свою основную работу завершённой.

## Этап 4. Новый Boss Orchestrator chat

Человек создаёт новый чат и передаёт туда bootstrap/контекст из Task Control Pack.

Boss Orchestrator стартует с чистого контекста.

Он не должен просить всю историю planning chat. Он должен работать по подготовленному пакету.

## Этап 5. Запуск первого B1-блока

Boss выбирает первый блок из block index и передаёт человеку или новому B1-чату соответствующий block handoff.

B1-чат начинает работу только по этому блоку.

## Этап 6. Выполнение блока

B1:

1. анализирует блок;
2. строит локальный план;
3. делит блок на 2–4 подзадачи;
4. назначает младших оркестраторов;
5. принимает отчёты;
6. проверяет результаты;
7. при необходимости запускает repair;
8. завершает блок.

## Этап 7. Возврат block report

B1 возвращает Boss Orchestrator-у отчёт блока.

Отчёт должен быть управленческим, а не сырым логом.

Он должен содержать:

- статус блока;
- что выполнено;
- какие подзадачи были;
- какие инструменты использовались;
- какие файлы изменены;
- что проверено;
- что не проверено;
- какие решения приняты;
- какие риски остались;
- что нужно сделать дальше.

## Этап 8. Boss проверяет блок

Boss не должен автоматически доверять B1.

Он должен:

- сверить block report с планом;
- проверить, не вышел ли блок за scope;
- оценить, закрыты ли критерии;
- понять, нужны ли repair/follow-up;
- обновить short plan;
- обновить journal;
- решить, запускать следующий блок или корректировать план.

## Этап 9. Корректировка плана при необходимости

Если B1 обнаружил новую реальность, Boss может изменить план.

Но изменение должно быть зафиксировано:

- причина;
- старое решение;
- новое решение;
- затронутые блоки;
- риск;
- статус принятия.

## Этап 10. Следующий блок

Boss запускает следующий B1-блок или создаёт новый блок, если план изменился.

Так продолжается, пока задача не будет завершена, заблокирована или переведена в новый planning cycle.

## Этап 11. Финализация задачи

После последнего блока Boss готовит final task report.

Он должен зафиксировать:

- что было сделано;
- какие блоки выполнены;
- какие решения приняты;
- какие файлы/области затронуты;
- какие проверки прошли;
- что осталось;
- какие риски;
- что нужно человеку проверить;
- можно ли считать задачу завершённой.

## Экономия Codex-контекста

## Что именно экономится

В этой системе важно экономить не все токены вообще, а именно главный Codex-контекст.

Можно сознательно тратить токены:

- на V1;
- на внешние чаты;
- на Kilo;
- на младших оркестраторов;
- на B1-чаты.

Если это разгружает Boss/главный Codex и позволяет ему читать только актуальные управленческие документы, система работает.

## Где появляется экономия

Экономия появляется на границах:

### 1. Planning Chat → Boss Orchestrator

Передаётся не вся история, а Task Control Pack.

### 2. Boss Orchestrator → B1

Передаётся не вся задача, а block handoff.

### 3. B1 → Junior Orchestrator

Передаётся не весь блок, а одна подзадача.

### 4. Kilo/External → Orchestrator

Возвращается не полный лог в чат, а отчёт + путь к полным материалам.

### 5. B1 → Boss

Возвращается block report, а не вся операционная переписка.

## Что ломает экономию

Экономию ломает:

- вставка полных V1/V2 ответов в Boss chat;
- вставка всех Kilo-логов в Boss chat;
- чтение всех `.md` “на всякий случай”;
- продолжение Planning Chat после завершения плана;
- использование одного B1-чата для нескольких блоков;
- отсутствие короткого рабочего плана;
- отсутствие границ между уровнями;
- просьбы “прочитай весь репозиторий” без необходимости;
- постоянное возвращение к полной истории.

## Правильное отношение к большим `.md`

Большие `.md` не плохи. Они нужны.

Они являются:

- архивом;
- доказательством;
- источником восстановления;
- источником аудита;
- базой для будущего анализа.

Но они не должны быть рабочим контекстом по умолчанию.

Рабочий контекст — это:

- короткий план;
- текущий статус;
- индекс блоков;
- отчёт блока;
- список решений;
- конкретный diff;
- конкретный target file;
- конкретный контракт, если он нужен.

---

## Контроль качества

Система не должна жертвовать качеством ради экономии.

Экономить надо не на проверке, а на лишнем контексте.

## Что обязательно проверять

В зависимости от задачи, на разных уровнях нужно проверять:

- изменённые файлы;
- соответствие scope;
- соответствие acceptance criteria;
- тесты;
- ошибки;
- safety boundaries;
- архитектурные последствия;
- влияние на будущие блоки;
- документы и журналы;
- не изменились ли запрещённые файлы;
- нет ли скрытых допущений.

## Кто что проверяет

### Младший оркестратор

Проверяет свою подзадачу и результат Kilo/внешнего инструмента.

### B1-оркестратор

Проверяет результат младших оркестраторов, целостность блока и соответствие block handoff.

### Boss Orchestrator

Проверяет block report, соответствие общему плану, влияние на следующие блоки и необходимость корректировки плана.

### Человек

Принимает финальные решения, особенно если:

- есть визуальный результат;
- есть UX;
- есть риск для проекта;
- есть изменение процесса;
- есть спорное архитектурное решение;
- есть неполная автоматическая проверка.

## Не доверять отчётам без проверки

Отчёт — это утверждение.

Проверка — это сверка утверждения с реальностью.

Если Kilo пишет “tests passed”, надо понимать:

- какие тесты;
- какой командой;
- в каком окружении;
- что не запускалось;
- не было ли warnings;
- соответствует ли это acceptance criteria.

Если внешний V2 пишет “approach looks safe”, надо понимать:

- видел ли он полный контекст;
- читал ли нужные файлы;
- были ли source boundaries;
- мог ли он проверить локальное состояние.

## Управление изменениями плана

План не должен быть статичным. Но изменение плана должно быть контролируемым.

## Когда можно менять план

План можно менять, если:

- блок выявил ошибочное допущение;
- появилась новая зависимость;
- Kilo обнаружил техническое ограничение;
- V2 нашёл серьёзную проблему;
- человек изменил цель;
- текущая последовательность блоков стала неэффективной;
- нужен дополнительный repair-block;
- часть блока стала не нужна.

## Как менять план правильно

Изменение должно фиксироваться как решение.

Нужно указать:

- что было раньше;
- что стало теперь;
- почему;
- какие блоки затронуты;
- есть ли риск;
- нужно ли согласование человека;
- какие документы обновлены.

## Где хранить изменения

Изменения должны попадать в:

- short plan;
- decision log;
- session journal;
- block index, если меняется структура блоков.

Подробный план можно не переписывать полностью каждый раз, если есть short plan как актуальная версия. Но если изменение крупное, подробный план тоже должен быть обновлён или помечен как устаревший в соответствующей части.

---

## Журналирование

Журнал нужен не для бюрократии, а для сохранения управляемого состояния.

## Что должен фиксировать session journal

Session journal должен отвечать на вопросы:

- какой блок был запущен;
- когда;
- с какой целью;
- какой отчёт вернулся;
- что Boss решил;
- какие проблемы возникли;
- какие repair были запущены;
- какие решения приняты;
- что стало следующим шагом.

## Что не нужно писать в журнал

В журнал не нужно вставлять:

- полные ответы внешних чатов;
- полные Kilo-логи;
- весь diff;
- весь raw transcript B1-чата;
- повторение всех старых решений.

Для этого есть отдельные полные источники. В журнале должны быть ссылки на них и управленческий итог.

## Decision log

Decision log должен быть ещё более чистым.

Он хранит только решения, а не процесс мышления.

Пример типов решений:

- принято использовать такой подход;
- отклонён такой подход;
- блок изменён;
- задача разделена;
- риск принят;
- риск требует human approval;
- часть работы перенесена в follow-up.

---

## Когда использовать полный процесс, а когда нет

Эта система не должна применяться к каждой мелкой задаче.

## Полный процесс оправдан, если задача:

- большая;
- многоэтапная;
- архитектурная;
- затрагивает много файлов;
- требует нескольких инструментов;
- имеет высокий риск ошибки;
- требует внешнего ревью;
- может менять workflow;
- должна оставить хороший след решений;
- важна для долгосрочного проекта.

## Полный процесс избыточен, если задача:

- маленькая;
- локальная;
- очевидная;
- затрагивает один файл;
- не требует архитектурного решения;
- легко проверяется;
- не требует внешнего мнения.

Для маленьких задач лучше использовать прямой путь: короткий handoff, Kilo, review, accept.

## Возможные риски системы

## Риск 1. Система станет слишком тяжёлой

Если каждый шаг требует слишком много документов, процесс может стать тяжелее самой задачи.

Решение:

- применять полный B1/Boss workflow только для больших задач;
- не плодить документы без необходимости;
- держать short plan коротким;
- хранить raw материалы отдельно;
- не использовать V1/V2/V3 “для галочки”.

## Риск 2. Потеря нюансов при переходе между чатами

Когда Planning Chat умирает, часть нюансов может потеряться.

Решение:

- делать качественный Task Control Pack;
- иметь подробный план как справочник;
- иметь decision log;
- явно фиксировать rejected options;
- хранить пути к полным источникам.

## Риск 3. B1-чат загрязняется

B1 может накопить много операционного шума.

Решение:

- один B1 = один блок;
- блок не должен быть слишком большим;
- B1 возвращает наверх только block report;
- raw материалы сохраняются отдельно.

## Риск 4. Младшие оркестраторы расходятся в стороны

Если младшим дать слишком широкие задачи, они начнут выполнять не то.

Решение:

- одна подзадача на младшего;
- чёткие границы;
- понятный ожидаемый отчёт;
- проверка B1;
- лимит repair loops.

## Риск 5. Boss начинает сам всё делать

Boss может втянуться в операционную работу и потерять роль.

Решение:

- Boss управляет блоками, а не выполняет все детали;
- для исполнения использует B1/Kilo/V1/V2/V3;
- читает полный источник только при необходимости;
- обновляет план и решения.

## Риск 6. Качество падает из-за экономии токенов

Если слишком сильно сжимать контекст, можно пропустить важные детали.

Решение:

- не экономить на проверке критичных diff/файлов;
- хранить полные источники;
- открывать full report при сомнениях;
- использовать V2 для рискованных результатов;
- фиксировать Not verified и Risks.

## Отношение к Obsidian

Obsidian в этой системе может быть полезен как человеческая карта смысла.

Но Obsidian не должен заменять:

- Git;
- `.ai` рабочие документы;
- реальные отчёты;
- журналы выполнения;
- source of truth по файлам.

Лучшее назначение Obsidian:

- карта процессов;
- карта решений;
- связи между V1/V2/V3/B1;
- объяснение “зачем это сделано”;
- обзор крупных задач;
- навигация по знаниям;
- заметки для человека.

Obsidian должен помогать человеку понимать систему, но не должен становиться ещё одним огромным контекстом, который Codex обязан читать целиком.

## Практическая модель уровней контекста

Можно мыслить систему как уровни.

## Уровень 0. Указатель

Минимальная информация:

- ID;
- статус;
- путь к документу;
- одна строка summary.

Используется для навигации.

## Уровень 1. Рабочая сводка

Короткий рабочий контекст:

- цель;
- статус;
- changed files;
- решения;
- риски;
- следующий шаг.

Это читает Boss или B1 по умолчанию.

## Уровень 2. Полный отчёт

Подробности:

- raw external answer;
- полный Kilo report;
- полный V2 review;
- полный block transcript;
- длинные проверки.

Читается только при необходимости.

## Уровень 3. Исторический архив

Старые обсуждения, rejected ideas, planning transcripts.

Нужны редко. По умолчанию не читаются.

## Важные границы

## Planning Chat

Может быть дорогим и исследовательским.  
Не должен жить после передачи плана.

## Boss Orchestrator

Должен управлять выполнением.  
Не должен читать всё и выполнять всё сам.

## B1

Должен выполнять один блок через младших оркестраторов.  
Не должен использоваться для нескольких блоков.

## Junior Orchestrator

Должен решать одну подзадачу.  
Не должен принимать решения всей задачи.

## Kilo

Должен выполнять локальную работу.  
Не должен считаться финальным источником истины без проверки.

## V1

Помогает думать.  
Не подтверждает выполнение.

## V2

Помогает ревьюить snapshot.  
Не заменяет локальную проверку.

## V3

Создаёт artifact packages.  
Не должен означать автоматический import без проверки.

---

## Что значит “задача завершена”

Задача считается завершённой не когда последний агент написал “готово”, а когда:

1. все запланированные блоки выполнены или явно отменены;
2. Boss принял block reports;
3. решения зафиксированы;
4. short plan отражает финальное состояние;
5. journal обновлён;
6. известные риски перечислены;
7. необходимые human checks выполнены или явно оставлены на человека;
8. нет незакрытых P0/P1 проблем;
9. финальный отчёт подготовлен;
10. человек может понять, что произошло, не читая всю историю.

## Главный смысл системы

Эта система не про бюрократию.

Она про то, чтобы AI-разработка стала управляемой.

Она позволяет:

- думать глубоко на этапе планирования;
- потом обрывать грязный контекст;
- передавать дальше только чистый управленческий пакет;
- выполнять большую задачу блоками;
- изолировать операционный шум;
- проверять слабых исполнителей;
- использовать сильные внешние чаты там, где они полезны;
- не заставлять Codex читать всё подряд;
- сохранять историю решений;
- корректировать план без хаоса;
- завершать задачу с понятным итогом.

Самая короткая формула:

```text
Планировочный чат создаёт смысл.
Boss Orchestrator управляет выполнением.
B1-чаты выполняют блоки.
Младшие оркестраторы решают подзадачи.
Kilo и внешние режимы являются инструментами.
Журналы и отчёты сохраняют правду.
Контекст обрывается на границах.
Наверх передаются решения, а не шум.
```

## Итоговое описание в одном абзаце

Иерархическая система AI-разработки устроена так: сначала отдельный planning chat без жёсткой экономии токенов глубоко исследует большую задачу, вызывает внешние мнения, формирует подробный план и создаёт Task Control Pack. Затем planning chat завершает свою роль, а новый Boss Orchestrator chat стартует с чистым контекстом и управляет выполнением по подготовленному пакету. Boss запускает отдельные B1 block chats, каждый из которых отвечает только за один блок, делит его на 2–4 подзадачи, нанимает младших оркестраторов, использует Kilo/V1/V2/V3 как инструменты, проверяет результаты и возвращает наверх block report. Boss проверяет block reports, обновляет short plan, journal и decision log, корректирует план при необходимости и запускает следующие блоки. Полные материалы сохраняются в `.md` файлах, но по умолчанию наверх передаются только управленческие сводки, решения, риски и ссылки на источники. Это позволяет сохранять качество проверки, не доверять слепо исполнителям, но при этом не засорять главный Codex-контекст длинной историей обсуждений и операционными логами.

---

## Финальная мысль

Самое ценное в этой системе — не наличие V1, V2, V3, Kilo или B1 по отдельности.

Самое ценное — границы.

Граница между планированием и выполнением.  
Граница между Boss и блоком.  
Граница между блоком и подзадачей.  
Граница между отчётом и доказательством.  
Граница между архивом и рабочим контекстом.  
Граница между делегацией и принятием.  

Если эти границы соблюдаются, система помогает строить большие проекты с AI без потери контроля.

Если границы исчезают, система превращается в длинный чат с большим количеством файлов.

Поэтому главный принцип можно сформулировать так:

> Каждый этап должен завершаться не бесконечным продолжением контекста, а аккуратной передачей управляемого состояния на следующий уровень.
