# Phase 5B — Pre-Kilo Package Review: подробная реализация шага

## 0. Назначение документа

Этот файл описывает только Phase 5B: проверку готового V3 artifact package до любого Kilo import. Здесь пакет оценивается как внешний объект, но ещё не применяется.

Этот документ написан как подробная практическая инструкция для rollout V3. Он не является короткой шпаргалкой и не заменяет исходные V3-контракты. Его задача — простым языком объяснить человеку, что именно делается на данном шаге, где проходит граница шага, какие ошибки опасны и как понять, что шаг выполнен.

Во всех формулировках ниже важно помнить базовую границу V3: внешний чат не имеет прямого доступа к репозиторию и не должен утверждать, что уже изменил файлы в Git. Внешний результат сначала существует как отдельный artifact package или как отдельное подготовительное действие. Запись в репозиторий, journal, Codex review и human acceptance — это отдельные части процесса, которые нельзя смешивать с текущим шагом.

## 1. Что это за шаг

Phase 5B — это защитный review-gate между внешним ZIP и будущим Kilo Notebook V3 import. На этом шаге человек или Codex-assisted reviewer открывает пакет, сверяет структуру, manifest, checksums, expected files, allowed paths, forbidden paths и содержание трёх markdown-файлов. Цель не в том, чтобы записать файлы в repo, а в том, чтобы решить: пакет можно допустить к следующему шагу, вернуть на revision или отклонить.

## 2. Зачем он нужен

Без 5B процесс был бы слишком резким: внешний чат создал ZIP, а человек сразу попытался бы импортировать его через Kilo. Для первого pilot это рискованно. Если в пакете неправильные пути, лишние файлы, плохой manifest или опасные формулировки, ошибка проявится слишком поздно. Pre-Kilo review позволяет остановить проблему до любого write-действия. Он особенно важен, потому что на раннем V3 rollout ещё нет полноценного scripted validator/staging слоя.

## 3. Почему он идёт именно сейчас

Этот шаг идёт после 5A, потому что сначала нужен реальный ZIP. Он идёт до 5C/5D import-действий, потому что Kilo не должен получать непроверенный пакет. Даже если package выглядит аккуратно, нельзя считать его безопасным до сверки с request и checksums. Сейчас правильный момент для ручного review: пакет уже существует, но repo ещё не должен быть затронут.

## 4. Что должно быть готово до начала шага

- Есть настоящий ZIP artifact package, созданный на Phase 5A.
- Доступен исходный V3 request, по которому пакет был создан.
- Известен список expected files и он содержит ровно три markdown-файла.
- Проверяющий понимает, что review не равен import и не равен acceptance.
- Есть безопасное место для просмотра/распаковки пакета вне рабочего repo root или без применения файлов.
- Проверяющий знает forbidden paths и current scope `workflow_docs`.


Эти prerequisites нужны не для бюрократии. Они защищают процесс от ситуации, когда участники вроде бы делают правильные действия, но на самом деле проверяют не тот объект, не ту фазу или не тот scope. Если хотя бы один базовый вход отсутствует, лучше остановиться и восстановить контекст, чем продолжать по памяти.

## 5. Что нельзя запускать раньше времени

На этом шаге есть жёсткие стоп-линии. Нельзя делать следующие действия:

- Распаковывать ZIP прямо в корень репозитория.
- Копировать файлы из `files/` в `.ai/v3/docs/`.
- Запускать Kilo Notebook V3 import.
- Создавать journal или journal draft.
- Обновлять `V3_navigation.md`.
- Делать Git commit.
- Автоматически исправлять пакет внутри архива и делать вид, что он таким был создан.
- Принимать Codex recommendation как final human verdict.
- Продолжать к import при несовпадающих checksums.


Если хочется сделать что-то из этого списка, это почти всегда признак, что человек уже перешёл к следующей фазе. В V3 это опасно: потом будет трудно понять, где именно появилась ошибка — в artifact generation, review, UI setup, import, journal или acceptance.

## 6. Пошаговая реализация


### Шаг 1. Зафиксировать объект проверки

Сначала нужно записать имя ZIP, V3 ID, task title, action, scope и expected files. Это можно сделать в обычных review notes. Не нужно создавать V3 journal: journal относится к import flow, а 5B — pre-import review.

### Шаг 2. Проверить, что это физический ZIP

Если вместо ZIP есть только текст ответа, набор markdown-блоков или ссылка на переписку, Phase 5B проводить нельзя. Сначала нужно получить настоящий artifact package.

### Шаг 3. Открыть дерево архива без применения

Нужно просмотреть список файлов ZIP. Ожидается корневая папка `V3-20260528-195750-phase5A-5C-deep-doc-pack/`, четыре control files и папка `files/`. Project files должны находиться только внутри `files/.ai/v3/docs/`.

### Шаг 4. Проверить control files

В корне должны быть `manifest.yaml`, `README_FOR_KILO.md`, `README_FOR_CODEX.md`, `checksums.sha256` и `files/`. Отсутствие любого элемента делает пакет неготовым к дальнейшему процессу.

### Шаг 5. Сверить manifest с request

Manifest нужно читать как паспорт пакета. Проверяются `v3_id`, `action: create`, `scope: workflow_docs`, current stage, allowed paths, forbidden paths, expected files и no repo access statement. Если manifest разрешает больше, чем request, это defect.

### Шаг 6. Сравнить три списка файлов

Нужно сравнить expected files из request, project_files из manifest и фактические файлы внутри `files/`. Все три списка должны совпадать. В текущей задаче project files должно быть ровно три.

### Шаг 7. Проверить forbidden paths

Даже один файл в `.ai/v3/contracts/`, `.ai/v3/prompts/`, `.ai/v3/templates/`, `.ai/v3/V3_navigation.md`, `scripts/`, `src/` или другом forbidden path должен блокировать дальнейшее движение пакета.

### Шаг 8. Сверить SHA-256

Для каждого project file нужно вычислить SHA-256 и сравнить со строкой в `checksums.sha256`. Любое несовпадение означает, что пакет нельзя считать целостным.

### Шаг 9. Прочитать README_FOR_KILO и README_FOR_CODEX

README_FOR_KILO не должен запускать import раньше времени. README_FOR_CODEX должен объяснять review, но не подменять human acceptance. Оба файла должны честно говорить, что текущий package ещё не изменил repo.

### Шаг 10. Проверить содержание трёх markdown-файлов

Нужно убедиться, что файл 5A говорит только об external artifact generation, файл 5B — только о pre-Kilo review, а файл 5C — только о UI setup. Все три файла должны быть подробными, русскоязычными и не должны притворяться, что import уже произошёл.

### Шаг 11. Сформулировать pre-Kilo verdict

Итогом 5B может быть `pass_to_next_step`, `revision_needed` или `reject`. Это не final acceptance. Это только решение, можно ли двигаться к следующему шагу.

## 7. Какие документы, решения и артефакты нужны

- ZIP package из Phase 5A.
- Исходный V3 request.
- V3 artifact package contract.
- V3 request contract.
- V3 storage policy.
- V3 acceptance policy.
- Список expected files.
- Review notes с результатами проверки.
- Pre-Kilo verdict: pass_to_next_step, revision_needed или reject.


Отдельно важно: если объект не назван в этом списке, его не нужно создавать “заодно”. Для раннего V3 rollout безопаснее сделать меньше, но точно в границах задачи, чем добавить полезный, но неразрешённый файл или действие.

## 8. Риски, проблемы, тупики и путаница


### Риск 1. Review случайно станет import

Самая опасная ошибка — распаковать пакет в repo root или начать копировать файлы. 5B должен смотреть на пакет, а не применять его.

### Риск 2. Проверить только структуру, но не содержание

ZIP может быть технически корректным, но тексты могут уходить в Phase 5D или утверждать, что repo уже изменён. Это тоже defect.

### Риск 3. Игнорировать checksums

Checksums нужны для целостности. Если их не сверить, нельзя уверенно сказать, что проверяется именно тот набор файлов, который был упакован.

### Риск 4. Принять лишний файл как полезное дополнение

Лишний project file нарушает expected_files. Ранний V3 workflow должен быть строгим, иначе позже будет сложно контролировать scope.

### Риск 5. Смешать Codex recommendation и human verdict

Codex может рекомендовать принять, но финальное решение остаётся за человеком. На 5B вообще речь только о pre-Kilo допуске.

## 9. Как реализовать шаг безопасно и прагматично

Практичный способ провести 5B — сделать два прохода. Первый проход механический: ZIP, дерево, manifest, checksums, paths, количество файлов. Второй проход смысловой: README, тексты трёх документов, отсутствие premature import claims. После этого фиксируется короткий pre-Kilo verdict. Важно ничего не чинить внутри пакета молча. Если пакет требует исправления, это revision, а не ручная подмена артефакта.

## 10. Как человеку проверить, что шаг выполнен

- [ ] Пакет был открыт для просмотра без применения к repo.
- [ ] Структура ZIP проверена.
- [ ] Control files присутствуют.
- [ ] Manifest прочитан и сравнен с request.
- [ ] Project files ровно три.
- [ ] Фактические файлы совпадают с expected files.
- [ ] Forbidden paths не нарушены.
- [ ] SHA-256 checksums совпадают.
- [ ] README_FOR_KILO не запускает import.
- [ ] README_FOR_CODEX сохраняет роль Codex как reviewer, а не final accepter.
- [ ] Три markdown-файла проверены по смыслу.
- [ ] Репозиторий не изменён.
- [ ] Зафиксирован pre-Kilo verdict.


Эта проверка специально написана человеческим языком. Её можно пройти без глубокого знания внутренних реализаций Kilo или будущих scripts/v3. Главное — не пропускать пункты, связанные с границами: нет repo write, нет преждевременного import, нет journal, если journal не входит в фазу.

## 11. Критерии приёмки шага

Шаг можно считать выполненным, если одновременно верны следующие пункты:

- Review выполнен до любого Kilo import.
- Проверка не изменила репозиторий.
- Manifest, checksums и project files согласованы.
- Нет лишних project files.
- Allowed paths и forbidden paths соблюдены.
- Содержание соответствует Phase 5A, 5B и 5C без дрейфа в 5D.
- Нет утверждений о выполненном import или repo write.
- Результат проверки явно зафиксирован.
- Человек понимает, можно ли допускать пакет к следующему шагу.


Если часть критериев выполнена, а часть нет, результат лучше считать не принятым, а требующим доработки. В V3 особенно важно не размывать понятие accepted: рекомендация, техническая готовность и человеческое финальное принятие — разные вещи.

## 12. Типовые ошибки, которые приведут к проблемам позже


1. **Распаковать ZIP в repo root**

   Это превращает review в несанкционированное применение файлов.


2. **Не сравнить manifest с request**

   Manifest может быть шире запроса, и это нужно поймать до import.


3. **Проверить только имена файлов**

   Содержание может нарушать фазовые границы, даже если имена правильные.


4. **Исправить package вручную без revision**

   Это ломает traceability и делает checksums/manifest недостоверными.


5. **Считать `pass_to_next_step` финальным accept**

   Это только предварительный допуск, а не human acceptance V3 result.


## 13. Выход шага

Выход Phase 5B — это review result по внешнему пакету. Правильная формула завершения: пакет проверен до Kilo, repo не изменён, journal не создан, есть понятный pre-Kilo verdict и решение, двигаться дальше или возвращать пакет на доработку.
