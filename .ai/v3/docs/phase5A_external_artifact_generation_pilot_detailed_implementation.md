# Phase 5A — External Artifact Generation Pilot: подробная реализация шага

## 0. Назначение документа

Этот файл описывает только Phase 5A: внешний pilot создания V3 artifact package. На этом шаге результатом должен быть настоящий ZIP-пакет, а не текстовый совет, не анализ и не заявленная запись в репозиторий.

Этот документ написан как подробная практическая инструкция для rollout V3. Он не является короткой шпаргалкой и не заменяет исходные V3-контракты. Его задача — простым языком объяснить человеку, что именно делается на данном шаге, где проходит граница шага, какие ошибки опасны и как понять, что шаг выполнен.

Во всех формулировках ниже важно помнить базовую границу V3: внешний чат не имеет прямого доступа к репозиторию и не должен утверждать, что уже изменил файлы в Git. Внешний результат сначала существует как отдельный artifact package или как отдельное подготовительное действие. Запись в репозиторий, journal, Codex review и human acceptance — это отдельные части процесса, которые нельзя смешивать с текущим шагом.

## 1. Что это за шаг

Phase 5A — это первый практический тест того, что внешний чат способен создать переносимый V3 artifact package. Смысл шага не в том, чтобы принять новые файлы в репозиторий, а в том, чтобы получить корректно упакованный ZIP: с корневой папкой, `manifest.yaml`, `README_FOR_KILO.md`, `README_FOR_CODEX.md`, `checksums.sha256` и проектными файлами только внутри `files/`. Внешний чат работает как генератор файлов, но не как участник с доступом к Git. Он читает V3 request, сверяется с указанными источниками и собирает пакет, который можно скачать и проверить. Если на этом шаге получится только обычный ответ в чате, Phase 5A не выполнена.

## 2. Зачем он нужен

Этот шаг нужен потому, что V3 отличается от V1 и V2 именно форматом результата. V1 даёт prompt-only анализ, V2 даёт bounded review, а V3 должен производить файлы. Пока не доказано, что внешний чат умеет стабильно возвращать package нужной структуры, бессмысленно проверять Kilo import: непонятно будет, где ошибка. Phase 5A изолирует самый первый риск — риск неправильного внешнего артефакта. Она позволяет поймать проблемы до того, как появятся Kilo, journal, Codex review и human verdict. Это особенно важно для первого pilot, где процесс ещё не hardened и где любая путаница может закрепиться как плохая привычка.

## 3. Почему он идёт именно сейчас

Phase 5A идёт после Phase 4, потому что к этому моменту V3 уже описан как отдельный workflow layer: есть README, navigation, контракты, шаблоны, prompt layer и manual setup guide для `Kilo Notebook V3`. Но полный operational cycle ещё не был доказан. Поэтому логичный следующий шаг — не сразу импортировать что-то через Kilo, а сначала проверить внешний artifact generation. На текущей стадии `/v3` shortcut и `scripts/v3/*` не являются обязательным runtime, а первый pilot ещё только начинается. 5A — безопасная фаза, потому что она не меняет repo и не требует запускать импорт.

## 4. Что должно быть готово до начала шага

- Есть структурированный V3 request с `v3_id`, `task_title`, `generated_by`, `action`, `scope`, контекстом, expected files, allowed paths, forbidden paths и acceptance criteria.
- `action` для текущего MVP установлен как `create`, без overwrite и delete.
- `scope` согласован с задачей; для данного пакета это `workflow_docs`.
- Список expected files заранее зафиксирован и содержит ровно три markdown-файла под `.ai/v3/docs/`.
- Внешний чат получил и прочитал V3 master plan, V3 README, V3 navigation, request contract, artifact package contract, storage policy, acceptance policy и manual Kilo Notebook V3 setup guide.
- Понимание границ подтверждено: нет repo access, нет прямой записи, нет Kilo import, нет journal, нет update `V3_navigation.md`.


Эти prerequisites нужны не для бюрократии. Они защищают процесс от ситуации, когда участники вроде бы делают правильные действия, но на самом деле проверяют не тот объект, не ту фазу или не тот scope. Если хотя бы один базовый вход отсутствует, лучше остановиться и восстановить контекст, чем продолжать по памяти.

## 5. Что нельзя запускать раньше времени

На этом шаге есть жёсткие стоп-линии. Нельзя делать следующие действия:

- Запускать Kilo Notebook V3 import.
- Распаковывать package в рабочий репозиторий как часть Phase 5A.
- Копировать файлы из `files/` в целевые repo paths.
- Создавать `.ai/v3/journals/*` или journal draft.
- Обновлять `.ai/v3/V3_navigation.md`.
- Создавать файлы вне `.ai/v3/docs/`.
- Добавлять четвёртый project file, даже если он кажется полезным.
- Писать, что репозиторий уже изменён.
- Писать, что Codex review или human acceptance уже выполнены.
- Переходить к Phase 5D Kilo Import Pilot.


Если хочется сделать что-то из этого списка, это почти всегда признак, что человек уже перешёл к следующей фазе. В V3 это опасно: потом будет трудно понять, где именно появилась ошибка — в artifact generation, review, UI setup, import, journal или acceptance.

## 6. Пошаговая реализация


### Шаг 1. Разобрать V3 request и выписать стоп-линии

Сначала нужно выделить фактические ограничения request: V3 ID, action, scope, allowed paths, forbidden paths, expected files и package format. После этого полезно явно записать стоп-линии: внешний чат только создаёт ZIP, не пишет в repo, не запускает Kilo, не создаёт journal, не обновляет navigation. Если в связанных документах встречаются более широкие формулировки, применяется более узкая граница текущего request.

### Шаг 2. Прочитать repo-grounded источники

Перед написанием файлов нужно свериться с текущим V3 canon. Особенно важны решения о том, что `kilo-notebook-v3` отделён от старого `kilo-notebook`, V3 не заменяет V1/V2, raw ZIP и staging относятся к local-only runtime, а финальное acceptance остаётся за человеком. Без этого документы могут получиться красивыми, но не совпасть с фактическим состоянием repo workflow.

### Шаг 3. Разделить содержание трёх будущих документов

Нужно заранее зафиксировать смысл каждого файла. Документ 5A объясняет только external artifact generation. Документ 5B объясняет только pre-Kilo review. Документ 5C объясняет только UI setup режима. Это предотвращает дрейф в Phase 5D и убирает повторение, когда все три файла начинают говорить одно и то же про импорт.

### Шаг 4. Создать project files только внутри `files/`

Внутри ZIP project files должны повторять repo-relative пути под папкой `files/`. Правильный пример: `files/.ai/v3/docs/phase5A_external_artifact_generation_pilot_detailed_implementation.md`. Неправильно класть `.ai/v3/docs/...` прямо в корень архива: тогда control files и project files смешиваются.

### Шаг 5. Проверить количество project files

В этом пакете должно быть ровно три project files. Не должно быть дополнительного README внутри `files/`, journal, navigation update, черновика, исходного prompt или служебного файла. Ранний V3 rollout лучше проверять на строгом маленьком наборе файлов.

### Шаг 6. Сформировать `manifest.yaml`

Manifest должен описывать пакет как транспортный контейнер: V3 ID, action, scope, allowed paths, forbidden paths, expected files, фактические project files, хэши и явное no repo access statement. Manifest не должен обещать import и не должен расширять разрешённые пути по сравнению с request.

### Шаг 7. Посчитать `checksums.sha256`

Для каждого файла внутри `files/` нужно вычислить SHA-256 и записать строку вида `<hash>  files/.ai/v3/docs/<name>.md`. Если после подсчёта файл менялся, checksums нужно пересчитать. Несовпадающий checksum должен блокировать дальнейшее использование пакета.

### Шаг 8. Добавить README для Kilo и Codex

`README_FOR_KILO.md` должен объяснять будущую безопасную проверку, но не запускать import сейчас. `README_FOR_CODEX.md` должен объяснять, как проверять пакет и какие признаки требуют revision или reject. Оба README должны сохранять границу: review и import не являются частью текущей внешней генерации.

### Шаг 9. Упаковать ZIP и проверить дерево

Финальный ZIP должен иметь корневую папку `V3-20260528-195750-phase5A-5C-deep-doc-pack/`. В корне только `manifest.yaml`, `README_FOR_KILO.md`, `README_FOR_CODEX.md`, `checksums.sha256` и `files/`. После упаковки нужно открыть список файлов архива и убедиться, что нет лишних или скрытых файлов.

## 7. Какие документы, решения и артефакты нужны

- Исходный V3 request.
- V3 master plan и текущий статус Phase 4.
- V3 request contract.
- V3 artifact package contract.
- V3 storage policy.
- V3 acceptance policy.
- Manual Kilo Notebook V3 setup guide.
- Три новых markdown-файла под `files/.ai/v3/docs/`.
- `manifest.yaml`, `README_FOR_KILO.md`, `README_FOR_CODEX.md`, `checksums.sha256`.
- Финальный ZIP artifact package.


Отдельно важно: если объект не назван в этом списке, его не нужно создавать “заодно”. Для раннего V3 rollout безопаснее сделать меньше, но точно в границах задачи, чем добавить полезный, но неразрешённый файл или действие.

## 8. Риски, проблемы, тупики и путаница


### Риск 1. Вернуть обычный текст вместо ZIP

Для V3 это не мелкое отклонение, а провал формата. Внешний ответ может быть полезным, но если нет реального ZIP с control files и `files/`, artifact generation pilot не доказан.

### Риск 2. Смешать external generation с import

Фразы “я записал файлы в repo”, “Kilo уже импортировал” или “journal создан” нарушают границу 5A. Такой пакет создаёт ложную историю и должен быть отправлен на доработку.

### Риск 3. Добавить лишний project file

Даже полезный четвёртый файл ломает expected_files. V3 держится на точных allowed paths и списках файлов, поэтому “добавил заодно” здесь опасно.

### Риск 4. Слишком абстрактное содержание

Задача требует подробных implementation docs. Короткая памятка не даст человеку понять, как безопасно пройти rollout, где остановиться и что проверять.

### Риск 5. Уйти в Phase 5D

Phase 5D — это будущий Kilo Import Pilot. В 5A можно упоминать его только как следующую границу, но нельзя описывать как уже выполняемое действие.

## 9. Как реализовать шаг безопасно и прагматично

Самый безопасный способ выполнить 5A — относиться к ZIP как к посылке, а не как к изменению репозитория. Все project files создаются во временной рабочей зоне, затем укладываются под `files/`, хэшируются и упаковываются. После этого пакет передаётся человеку. Никакие repo paths не трогаются. Если возникает желание “сразу применить”, это нужно остановить и перенести в будущую фазу с отдельным разрешением. Хороший 5A-пакет должен быть скучным, предсказуемым и легко проверяемым: три файла, один manifest, один checksums-файл, два README и никаких сюрпризов.

## 10. Как человеку проверить, что шаг выполнен

- [ ] ZIP физически существует и скачивается как файл.
- [ ] В ZIP есть корневая папка с V3 ID.
- [ ] В корне есть `manifest.yaml`, `README_FOR_KILO.md`, `README_FOR_CODEX.md`, `checksums.sha256` и `files/`.
- [ ] `files/` содержит ровно три project files.
- [ ] Все три project files лежат под `files/.ai/v3/docs/`.
- [ ] Нет project files вне `files/`.
- [ ] Нет файлов в forbidden paths.
- [ ] `manifest.yaml` согласован с request.
- [ ] `checksums.sha256` совпадает с содержимым project files.
- [ ] Тексты не утверждают, что repo уже изменён или import уже выполнен.


Эта проверка специально написана человеческим языком. Её можно пройти без глубокого знания внутренних реализаций Kilo или будущих scripts/v3. Главное — не пропускать пункты, связанные с границами: нет repo write, нет преждевременного import, нет journal, если journal не входит в фазу.

## 11. Критерии приёмки шага

Шаг можно считать выполненным, если одновременно верны следующие пункты:

- Создан настоящий ZIP artifact package.
- Package structure соответствует V3 artifact package contract.
- Project files ровно три и все ожидаемые.
- Все project files написаны на русском языке и достаточно подробны.
- Phase 5A file не прыгает в Kilo import.
- Package не создаёт journal и не обновляет `V3_navigation.md`.
- Allowed paths и forbidden paths соблюдены.
- No repo access statement сохранён.
- Финальный результат передан как ZIP, а не как утверждение о repo changes.


Если часть критериев выполнена, а часть нет, результат лучше считать не принятым, а требующим доработки. В V3 особенно важно не размывать понятие accepted: рекомендация, техническая готовность и человеческое финальное принятие — разные вещи.

## 12. Типовые ошибки, которые приведут к проблемам позже


1. **Заявить, что репозиторий изменён**

   Внешний чат не имеет такого доступа. Это нарушает доверие к workflow и делает дальнейший review подозрительным.


2. **Положить project files в корень архива**

   Project files должны быть только под `files/`, иначе пакет нельзя безопасно интерпретировать.


3. **Не пересчитать checksums после правок**

   Checksums станут недостоверными, и pre-Kilo review должен будет остановить пакет.


4. **Добавить journal draft**

   Journal появляется только в import flow, а не на внешней генерации.


5. **Сделать файл 5A слишком коротким**

   Пользователь просил полноценный разбор реализации, а не краткую памятку.


## 13. Выход шага

Выход Phase 5A — это ZIP artifact package. Правильная формула завершения: внешний пакет создан, структура проверяема, project files находятся внутри `files/`, прямых изменений репозитория нет, Kilo import не запускался, journal не создавался.
