# Phase 5C — Kilo Notebook V3 UI Setup: подробная реализация шага

## 0. Назначение документа

Этот файл описывает только Phase 5C: ручную настройку и проверку режима `Kilo Notebook V3` в UI Kilo Code. Это не импорт ZIP и не применение файлов.

Этот документ написан как подробная практическая инструкция для rollout V3. Он не является короткой шпаргалкой и не заменяет исходные V3-контракты. Его задача — простым языком объяснить человеку, что именно делается на данном шаге, где проходит граница шага, какие ошибки опасны и как понять, что шаг выполнен.

Во всех формулировках ниже важно помнить базовую границу V3: внешний чат не имеет прямого доступа к репозиторию и не должен утверждать, что уже изменил файлы в Git. Внешний результат сначала существует как отдельный artifact package или как отдельное подготовительное действие. Запись в репозиторий, journal, Codex review и human acceptance — это отдельные части процесса, которые нельзя смешивать с текущим шагом.

## 1. Что это за шаг

Phase 5C — это подготовка UI/runtime уровня: человек должен убедиться, что в Kilo Code существует отдельный режим `Kilo Notebook V3` с internal value `kilo-notebook-v3`, и что он не перепутан со старым `Kilo Notebook`. На этом шаге проверяется видимость режима, правильное название, правильная роль и понимание границ. Сам artifact package не импортируется.

## 2. Зачем он нужен

Repo-level документы могут уже говорить, что режим существует, но это не гарантирует, что человек реально видит его в интерфейсе Kilo Code. Если перед первым import pilot режим не настроить, можно случайно выбрать старый `Kilo Notebook`, который относится к `/v1-only`, или другой режим. Phase 5C нужна, чтобы новый V3 route не остался только на бумаге и чтобы future import запускался в правильной UI-рамке.

## 3. Почему он идёт именно сейчас

Этот шаг идёт после подготовки и проверки пакета, но до Kilo Import Pilot. Логика rollout такая: 5A создаёт ZIP, 5B проверяет ZIP, 5C готовит UI mode, а только отдельный будущий шаг может запускать import. Сейчас правильный момент проверить UI, потому что процесс уже достаточно конкретный, но запись ещё не началась. При этом `/v3` shortcut и `scripts/v3/*` не требуются: setup выполняется вручную.

## 4. Что должно быть готово до начала шага

- Repo-level V3 canon уже описывает `kilo-notebook-v3` и UI-имя `Kilo Notebook V3`.
- Есть manual setup guide для Kilo Notebook V3.
- Человек понимает различие между V1, V2 и V3.
- Человек понимает, что старый `Kilo Notebook` остаётся `/v1-only`.
- Доступен интерфейс Kilo Code в VS Code.
- Есть отдельное решение, что на этом шаге выполняется только UI setup, без import.


Эти prerequisites нужны не для бюрократии. Они защищают процесс от ситуации, когда участники вроде бы делают правильные действия, но на самом деле проверяют не тот объект, не ту фазу или не тот scope. Если хотя бы один базовый вход отсутствует, лучше остановиться и восстановить контекст, чем продолжать по памяти.

## 5. Что нельзя запускать раньше времени

На этом шаге есть жёсткие стоп-линии. Нельзя делать следующие действия:

- Передавать ZIP в Kilo для фактического import.
- Распаковывать artifact package в staging как часть import.
- Копировать project files в repo paths.
- Создавать journal draft.
- Обновлять `V3_navigation.md`.
- Использовать старый `Kilo Notebook` как V3 mode.
- Использовать `Kilo Recorder` как V3 import mode.
- Создавать `/v3` shortcut без отдельной задачи.
- Создавать `scripts/v3/*`.
- Запускать auto-apply или Git commit.


Если хочется сделать что-то из этого списка, это почти всегда признак, что человек уже перешёл к следующей фазе. В V3 это опасно: потом будет трудно понять, где именно появилась ошибка — в artifact generation, review, UI setup, import, journal или acceptance.

## 6. Пошаговая реализация


### Шаг 1. Открыть Kilo Code в VS Code

Настройка выполняется в интерфейсе, где позже предполагается запускать V3-задачи. Но на этом шаге не нужно прикладывать ZIP и не нужно отправлять import-задание.

### Шаг 2. Найти список или настройки Kilo modes

Конкретный UI может отличаться, поэтому важно не место кнопки, а результат: должен быть отдельный mode для V3. Нельзя переименовать старый mode и считать задачу выполненной.

### Шаг 3. Добавить или проверить `Kilo Notebook V3`

Нужно использовать internal value `kilo-notebook-v3`, UI name `Kilo Notebook V3` и recommended role `Notebook Agent`. Описание режима должно говорить о безопасной проверке V3 artifact package, manifest, checksums, allowed/forbidden paths, journal flow и Codex review, но не запускать импорт прямо при setup.

### Шаг 4. Убедиться, что старый `Kilo Notebook` остался отдельным

Старый `Kilo Notebook` относится к `/v1-only`. Он не должен быть переиспользован для V3. В UI должны различаться `Kilo Notebook` и `Kilo Notebook V3`, если оба доступны.

### Шаг 5. Убедиться, что `Kilo Recorder` не используется для V3

`Kilo Recorder` относится к другому route. Его нельзя использовать как import mode для V3 artifact package.

### Шаг 6. Связать режим с V3 mode prompt

Если UI позволяет добавить mode prompt, нужно использовать V3 prompt для `kilo-notebook-v3`. Если UI не позволяет, prompt должен быть подготовлен как обязательный текст будущего запуска. На 5C сам запуск не выполняется.

### Шаг 7. Проверить видимость режима после сохранения

После настройки нужно убедиться, что `Kilo Notebook V3` реально виден в списке modes. Если возможно, полезно закрыть и снова открыть список или обновить UI.

### Шаг 8. Сделать только безопасную dry-check проверку

Допустимо выбрать режим и убедиться, что UI показывает правильный mode. Нельзя отправлять задание на импорт и нельзя прикладывать package для применения. Если UI непредсказуем, лучше вообще не отправлять сообщение.

### Шаг 9. Зафиксировать результат setup

Достаточно рабочей заметки: режим добавлен, internal value проверен, старый notebook не изменён, recorder не затронут, import не запускался. V3 journal на этом шаге не создаётся.

### Шаг 10. Остановиться перед import

После UI setup нужно остановиться. Любой фактический import — отдельная фаза с отдельным разрешением и отдельными проверками.

## 7. Какие документы, решения и артефакты нужны

- Manual Kilo Notebook V3 setup guide.
- V3 README и V3 navigation для проверки текущего статуса.
- V3 mode prompt для `kilo-notebook-v3`.
- Доступ к Kilo Code UI.
- Рабочая заметка о результате настройки.


Отдельно важно: если объект не назван в этом списке, его не нужно создавать “заодно”. Для раннего V3 rollout безопаснее сделать меньше, но точно в границах задачи, чем добавить полезный, но неразрешённый файл или действие.

## 8. Риски, проблемы, тупики и путаница


### Риск 1. Переиспользовать старый `Kilo Notebook`

Это главный риск. Старый mode имеет другой маршрут и не должен становиться V3 “по совместительству”.

### Риск 2. Запустить import во время setup

Желание сразу попробовать понятно, но оно смешает 5C с будущей 5D. На 5C проверяется только режим.

### Риск 3. Неверный internal value

UI name может выглядеть правильно, но если internal value не `kilo-notebook-v3`, настройка не соответствует canon.

### Риск 4. Ждать `/v3` shortcut

На текущем этапе shortcut не обязателен и не должен создаваться в рамках 5C. Режим выбирается вручную.

### Риск 5. Ожидать `scripts/v3/*`

Скрипты относятся к более поздней фазе. 5C — ручной UI setup, а не scripted support.

## 9. Как реализовать шаг безопасно и прагматично

Безопасная реализация 5C минимальна: добавить или проверить режим, убедиться в правильном internal value, отличить его от старых режимов, связать с mode prompt и остановиться. Не нужно “заодно” импортировать пакет, создавать journal или обновлять navigation. Чем меньше действий на 5C, тем чище будет последующий pilot: если позже что-то пойдёт не так, будет понятно, что UI setup был отдельной проверенной стадией.

## 10. Как человеку проверить, что шаг выполнен

- [ ] В UI виден режим `Kilo Notebook V3`.
- [ ] Internal mode value равен `kilo-notebook-v3`.
- [ ] Рекомендуемая role связана с `Notebook Agent`.
- [ ] Старый `Kilo Notebook` не переиспользован и остаётся отдельным `/v1-only` режимом.
- [ ] `Kilo Recorder` не используется для V3.
- [ ] Mode prompt для V3 доступен для будущего запуска.
- [ ] ZIP не импортировался.
- [ ] Файлы не копировались в repo.
- [ ] Journal не создавался.
- [ ] `V3_navigation.md` не обновлялся.
- [ ] Итог setup зафиксирован без repo write.


Эта проверка специально написана человеческим языком. Её можно пройти без глубокого знания внутренних реализаций Kilo или будущих scripts/v3. Главное — не пропускать пункты, связанные с границами: нет repo write, нет преждевременного import, нет journal, если journal не входит в фазу.

## 11. Критерии приёмки шага

Шаг можно считать выполненным, если одновременно верны следующие пункты:

- Режим `Kilo Notebook V3` существует в UI.
- Internal value точно `kilo-notebook-v3`.
- Режим не конфликтует со старым `Kilo Notebook`.
- Человек понимает назначение режима и отличие от V1/V2/R1.
- UI setup выполнен без import.
- Repo не изменён.
- Journal не создан.
- Следующий шаг не начат без отдельного разрешения.


Если часть критериев выполнена, а часть нет, результат лучше считать не принятым, а требующим доработки. В V3 особенно важно не размывать понятие accepted: рекомендация, техническая готовность и человеческое финальное принятие — разные вещи.

## 12. Типовые ошибки, которые приведут к проблемам позже


1. **Указать красивое UI-имя, но неправильный internal value**

   Внешне режим может выглядеть верно, но process canon будет нарушен.


2. **Использовать старый `Kilo Notebook`**

   Это смешивает V1 и V3 и создаёт неправильный operational route.


3. **Запустить package import во время UI проверки**

   Такой шаг уже относится к будущему import pilot, а не к setup.


4. **Создать journal на 5C**

   Journal появляется в import flow, не в UI setup.


5. **Считать отсутствие scripts проблемой**

   На текущей фазе ручная настройка ожидаема, scripts появятся позже.


## 13. Выход шага

Выход Phase 5C — подтверждённая готовность UI-режима `Kilo Notebook V3`. Правильная формула завершения: режим виден и настроен, internal value проверен, старые режимы не смешаны, import не запускался, repo не изменялся.
