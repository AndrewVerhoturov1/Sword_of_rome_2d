## Ключевые выводы внешнего review

### Гипотеза подтверждена

Внешний ревьюер подтверждает: [`handleRotateMouseDown`](table-sandbox/src/editor/EditorSurface.tsx:618-619) некорректно вычисляет world-space центр вращения. При `scale ≠ 1` ошибка составляет `naturalWidth * (scale - 1) / 2` по X и аналогично по Y.

### Root cause

Локальный drift между двумя способами вычисления одной и той же точки:
- Канонический путь через `mapLocalToWorld(cx, cy, underlay)` → возвращает `(offsetX + cx, offsetY + cy)` для центра.
- `handleRotateMouseDown` вычисляет вручную `offsetX + naturalWidth * scale / 2` — лишнее умножение на scale.

### Рекомендованный correction path

Использовать `mapLocalToWorld` в `handleRotateMouseDown` для консистентности с `handleScaleHandleMouseDown`:

```ts
const originUnderlay = { ...underlay };
const center = mapLocalToWorld(
  originUnderlay.naturalWidth / 2,
  originUnderlay.naturalHeight / 2,
  originUnderlay
);
```

Прямая формула `offsetX + naturalWidth/2` тоже корректна, но `mapLocalToWorld` предпочтительнее для консистентности.

### Риски

- Не менять `mapLocalToWorld` — он корректен.
- Не расширять scope за пределы rotate center.
- Ожидается изменение поведения при rotate на scale ≠ 1 (вращение вокруг истинного центра) — нужен human visual check.

### Принятое решение

Применяем fix через `mapLocalToWorld` для консистентности, без изменения других функций.

## Human Check после реализации

suggested — проверить rotate underlay при scale ≠ 1 (например 0.5 или 2). Ожидается вращение вокруг геометрического центра изображения, а не смещённой точки. Также проверить scale=1 (без регрессии) и Shift-snap (15°).
