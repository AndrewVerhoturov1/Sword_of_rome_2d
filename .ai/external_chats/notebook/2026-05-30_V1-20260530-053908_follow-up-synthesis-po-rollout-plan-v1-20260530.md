### Confirmed from central docs

Central `/v1` rules require honest source boundaries: external chat is not authority over the local repo, `fully read` is allowed only for actually opened/read material, and repo/workflow answers must separate central docs, provided excerpts, and unverified claims. That is why this answer treats pinned project raw links and provided excerpts as the source, not local shell/git access. ([GitHub][1])

Central navigation defines `ai-workflow-core` as reusable core with stable rules, prompts, templates, scripts, and external-chat materials, but explicitly not as consumer repo runtime/history/project state. So central core can justify reusable B1 primitives, but cannot prove that this specific repo has successfully run B1. ([GitHub][2])

Central and project B1 materials agree on the key lifecycle: `Block Plan + Context Pack -> Block Orchestrator Package -> Junior Block Orchestrator -> Executor Run -> Block Report -> Lead/Boss Review`. The `create_block_plan` prompt makes `/b1` planning-only, requires `2–4` planned agent calls, and separates planned work from contingency/repair runs. ([GitHub][3])

The package gate is real, not decorative: `Main Execution Orchestrator Chat` must not directly create a Kilo handoff or external package for block execution; first it must create a `Block Orchestrator Package` and use it to hire the junior orchestrator. ([GitHub][4])

Project-local `codex_orchestrator.md` is even clearer: `Strategist`, `Main Execution Orchestrator`, `Block Orchestrator Chat`, and `Executor Run` are separate roles; repo lookup/test discovery inside a block is substantive work and must go through Kilo; skipping the package gate is `blocked`. ([GitHub][5])

The review gate already exists: `review_block_report.md` requires checking the Block Report against Block Plan, Context Pack, changed files, Kilo reports, allowed scope, acceptance criteria, agent-first evidence, and package-gate usage. ([GitHub][6])

#### What to keep from the current plan

Оставить нужно не весь план как есть, а его правильную ось:

1. **Project-local rollout first.** Сначала доказывать систему внутри `Sword_of_rome_2d`, а не сразу тащить всё в `ai-workflow-core`.
2. **Task Control Pack + Boss bootstrap before B1 execution.** Без этого B1-блоки снова будут отдельными кусками без владельца.
3. **Первый docs-only smoke pilot.** Он полезен, но только как проверка форм и маршрута, не как доказательство всей системы.
4. **Block Orchestrator Package gate.** Это один из самых сильных уже существующих элементов scaffold.
5. **Promotion to central core late.** В central core переносить только то, что пережило реальные pilot cycles.
6. **Не делать новый audit.** B1 audit уже говорит: scaffold сильный, published evidence реального B1-run не найдено.

#### What to change in the current plan

Главная правка: план должен перестать строить "полный офис подпроекта" до первого реального рабочего дня.

Что поменять:

1. **Добавить Pre-Phase ID decision.** До создания документов решить: какой ID считается главным. Нельзя начинать `BLOCKS_INDEX.md`, `BLOCK_REPORT.md`, `BOSS_REVIEW.md`, если непонятно, `BLOCK-001` это или `B1-001`.

2. **Разделить Phase 1 на 1a и 1b.**
   Phase 1a — только минимальные canonical docs, нужные для первого pilot.
   Phase 1b — расширение структуры после первого pilot, когда станет понятно, какие поля реально нужны.

3. **Сократить стартовый skeleton.** Не создавать заранее полный набор `SESSION_JOURNAL`, `Block reports`, `Kilo reports`, `Final report`, пустые папки под всё будущее. Runtime/history artifacts появляются только во время или после исполнения.

4. **Сделать Package gate настоящим pass/fail gate.** Перед запуском Junior Orchestrator Boss должен проверить package. Если package не содержит scope, boundary, recommended agent path, stop conditions, planned sequence, human checkpoints и fail-fast gates — блок не стартует.

5. **Добавить explicit pilot exit criteria.** Успех pilot — это не "файлы созданы", а "цепочка прошла и оставила evidence".

6. **Добавить второй pilot до hardening/promotion.** Docs-only smoke block проверяет формы. Второй non-trivial block проверяет роль Junior Orchestrator под реальной нагрузкой.

7. **Связку V1/V2/V3/Kilo делать после первых evidence.** Не мигрировать старую историю задним числом и не строить большую metadata-систему до понимания, какие ссылки реально нужны.

#### ID decision

Решение: **выбрать один canonical block ID: `BLOCK-NNN_slug`.**

Не держать `BLOCK-001` и `B1-001` как два равноправных ID одной сущности. Это создаст путаницу в `BLOCKS_INDEX`, `BLOCK_REPORT`, `BOSS_REVIEW`, future validators и V1/V2/V3/Kilo metadata.

Правило:

```text
Subproject ID: SP-YYYYMMDD-slug
Canonical Block ID: BLOCK-NNN_slug
B1: название слоя/режима/типа workflow, не ID блока
B1 Run ID: optional runtime attempt ID only, e.g. B1RUN-20260530-001
```

В canonical docs писать:

```text
Block ID: BLOCK-001-task-control-pack-smoke
Layer: B1
Subproject ID: SP-20260530-b1-boss-rollout
```

В runtime docs можно добавить:

```text
B1 Run ID: B1RUN-20260530-001
Attempt: 001
```

Но `B1RUN` не должен заменять `BLOCK-001` и не должен быть главным ключом в index.
