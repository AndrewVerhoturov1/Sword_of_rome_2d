Usage Rules
1. V1-Синтез

Использовать, когда нужна внешняя мысль, критика, naming, план, объяснение или черновик, а потом субагент должен собрать это в нормальный документ.

Не использовать, когда нужны локальные repo-факты, file edits, diff-review, тесты или отладка. Project rules отдельно указывают, что External Web Chat не имеет доступа к файловой системе repo и не является источником repo-фактов без локальной проверки.

2. V3-Ревью

Использовать для важных документов и артефактов, когда полезно сначала получить сильный V3-драфт/package, а потом дать субагенту проверить, слегка поправить и подготовить к локальному использованию.

Не использовать, если нет реального V3-драфта/package, если нужно большое переписывание, или если задача фактически требует локальных тестов/отладки.

Важно: в документации надо явно написать, что V3-Ревью — это рабочее короткое название для данной связки, а не новое значение /v3. В project rules /v3 описан как explicit import-entry route для уже существующего V3 artifact package, не как новый Kilo mode или новый Agent kind.

3. Kilo

Использовать, когда нужна локальная repo-grounded работа:

изменить файлы;
проверить diff;
запустить тесты;
отладить;
подтвердить, что repo реально в нужном состоянии;
проверить локальные факты.

Kilo не переименовывать. В общей линейке он должен быть описан как локальный инструмент субагента для repo-grounded исполнения, проверки и отладки.

Project rules подтверждают, что Codex — оркестратор, а Kilo Code — исполнитель маленьких изолированных задач; также там перечислены допустимые Kilo modes, включая runner/debugger/verifier/notebook variants.

4. Субагент-микро

Использовать только как маленькое исключение, когда задача настолько мелкая, что запускать V1/V3/Kilo тяжелее, чем сделать её напрямую.

Примеры:

слегка поправить формулировку;
сократить название;
составить маленький список;
привести одну таблицу к нормальному виду;
сделать маленькую редакторскую правку внутри уже понятного документа.

Не использовать для важных документов, repo-фактов, архитектурных решений, file edits, тестов или задач с риском.

# Инструменты субагента

## Status

Draft.

Этот документ описывает короткий набор инструментов, которыми может пользоваться субагент внутри B1/block workflow.

Это не общий справочник всех инструментов проекта и не новый routing-документ для всего repo.

## Scope

Эти инструменты используются только внутри процесса:

```text
Subproject -> Block Plan -> Block Orchestrator -> Subagent Tasks
```

Инструмент выбирается отдельно для конкретной задачи блока.

Выбор планирует старший оркестратор блока / Block Orchestrator.

Субагент использует выбранный инструмент для выполнения задачи.

Main Rule

Старший оркестратор блока не обязан выбирать один инструмент для всего блока.

Он выбирает инструмент отдельно для каждой задачи:

если нужна внешняя мысль и сборка результата — V1-Синтез;
если нужен сильный V3-артефакт и проверка — V3-Ревью;
если нужна локальная работа с repo — Kilo;
если задача совсем маленькая — Субагент-микро.
Инструменты
Инструмент	Старое название	Кто использует	Кто планирует	Для чего
V1-Синтез	V1 + subagent synthesis	субагент	старший оркестратор блока	внешний V1 даёт ответ, субагент собирает итог
V3-Ревью	V3 + subagent verify/minor fix	субагент	старший оркестратор блока	V3 даёт сильный артефакт, субагент проверяет и слегка правит
Kilo	Kilo	субагент / локальный исполнитель	старший оркестратор блока	локальная repo-grounded работа, diff, тесты, отладка, проверка
Субагент-микро	Subagent direct work	сам субагент	старший оркестратор блока	очень маленькая задача без отдельного инструмента
1. V1-Синтез
Кратко

V1-Синтез — это связка:

внешний V1-ответ -> субагент собирает итоговый документ
Когда использовать

Использовать, если задача про:

naming;
critique;
план;
объяснение;
документационный design;
second opinion;
черновик текста;
workflow clarity.
Когда не использовать

Не использовать, если задача требует:

локального чтения repo;
file edits;
diff-review;
тестов;
отладки;
подтверждения локальных repo-фактов.
Сильные стороны
хорошо даёт идеи и варианты;
помогает сделать naming/design;
подходит для внешней критики;
дешевле и проще, чем полноценный artifact workflow.
Риски
внешний ответ не является repo-authority;
результат нужно проверять и собирать;
нельзя принимать repo-факты без локальной проверки.
Чем отличается

V1-Синтез — это не готовый artifact package. Это внешний ответ, который субагент превращает в полезный итог.

2. V3-Ревью
Кратко

V3-Ревью — это связка:

V3 делает сильный драфт/артефакт -> субагент проверяет и минимально правит
Когда использовать

Использовать, если:

задача важная;
нужен сильный первый драфт;
нужен готовый markdown/html/zip/artifact;
субагент должен проверить, чуть поправить или перенести результат;
baseline preference для важной документации = V3.
Когда не использовать

Не использовать, если:

V3-драфта или artifact package ещё нет;
требуется большая перепись;
задача требует локальной отладки;
задача требует тестов;
субагент не может проверить результат.
Сильные стороны
лучший baseline для важных документов;
даёт сильный внешний драфт;
субагент не пишет всё с нуля, а проверяет и доводит;
хорошо подходит для artifact-producing workflow.
Риски
"минимальная правка" может незаметно стать большой переписью;
нельзя автоматически принимать claims из артефакта;
нужно не путать V3-Ревью с /v3 import-entry route.
Чем отличается

V3-Ревью отличается от V1-Синтез тем, что на входе ожидается более готовый артефакт или сильный драфт, а не просто внешний анализ.

3. Kilo
Кратко

Kilo — это локальный repo-grounded инструмент.

Название Kilo не менять.

В этой линейке Kilo означает:

локальное исполнение / отладка / проверка
Когда использовать

Использовать, если нужно:

работать с локальными файлами;
менять файлы;
проверять diff;
запускать тесты;
отлаживать;
проверять, что repo реально находится в нужном состоянии;
выполнять verifier-задачи.
Когда не использовать

Не использовать, если:

задача чисто про идеи или naming;
нет необходимости в локальном repo;
задача слишком маленькая и проще сделать через Субагент-микро;
нужен сильный внешний драфт для документации, где лучше сначала использовать V3-Ревью.
Сильные стороны
имеет локальную repo-grounded роль;
подходит для file edits;
подходит для tests/diff/debug;
хорошо работает как runner/debugger/verifier.
Риски
тяжелее, чем маленькая прямая правка;
требует правильного handoff;
нельзя путать Kilo mode и task role;
нельзя использовать как декоративный agent run без реального результата.
Чем отличается

Kilo — это не внешний reasoning route. Это локальная работа с repo.

4. Субагент-микро
Кратко

Субагент-микро — это исключение для очень маленькой задачи.

субагент делает сам, без V1/V3/Kilo
Когда использовать

Использовать, если:

задача очень маленькая;
нет repo-risk;
нет необходимости в тестах;
нет необходимости в diff-review;
запуск отдельного инструмента тяжелее самой задачи.
Примеры
сократить название;
поправить одну формулировку;
сделать маленькую таблицу;
привести короткий текст к нужному стилю;
уточнить один пункт в уже понятном документе.
Когда не использовать

Не использовать, если:

задача важная;
задача влияет на repo-файлы с риском;
нужны локальные факты;
нужны тесты;
нужна отладка;
нужно архитектурное решение;
нужна проверка внешнего артефакта.
Сильные стороны
быстро;
не создаёт лишний overhead;
удобно для мелких правок.
Риски
легко начать злоупотреблять;
может обойти нормальный workflow;
слабая traceability;
не должно становиться default-route.
Чем отличается

Субагент-микро — это не полноценный инструмент наравне с V1/V3/Kilo. Это маленькое исключение.

Selection Cheat Sheet
Ситуация	Инструмент
Нужна внешняя мысль, naming, critique	V1-Синтез
Нужен важный документ или сильный artifact draft	V3-Ревью
Нужны локальные файлы, diff, тесты, отладка	Kilo
Нужно сделать совсем маленькую правку	Субагент-микро
Recommended Block Plan Field

В block/task docs можно фиксировать выбор так:

Subagent Tool: V3-Ревью
Reason: важный workflow-документ; нужен сильный V3-драфт и проверка субагентом.

Или:

Subagent Tool: Kilo
Reason: задача требует локальной проверки файлов и diff.

Или:

Subagent Tool: V1-Синтез
Reason: нужна внешняя критика и последующая сборка итогового design.

Или:

Subagent Tool: Субагент-микро
Reason: задача мелкая; запуск отдельного инструмента тяжелее самой правки.
Hard Boundaries
Не добавлять новые инструменты без отдельного решения.
Не использовать этот документ как общий routing-документ всего repo.
Не считать Субагент-микро default route.
Не переименовывать Kilo.
Не путать V3-Ревью с /v3 import-entry route.
Не принимать внешние ответы как repo-факты без локальной проверки.
Для каждой задачи блока выбирать один основной инструмент.
Если инструмент выбрать невозможно честно — вернуть Blocked / Clarification Request.
Out of Scope

Этот документ пока не описывает:

полный /r1 published external route;
полный /v2 external senior review;
весь V3 import contract;
Kilo Recorder;
Kilo Notebook;
глобальную маршрутизацию всего repo;
product-code execution policy вне B1/block workflow.
