([GitHub][11])

### Main assessment

The V2 docs set is now **much stronger** and close to usable. The core concept is now coherent: V2 is manual, local to this repo, not `/v1`, not `/r1`, not a new Kilo mode, not recorder/notebook expansion, and not an automatic code-fix system. ([GitHub][11])

But before verifier/pilot, I would still fix several wording and template issues. Most are not architectural; they are precision problems that could confuse Kilo during a real first pilot.

### Contract conflicts

1. **Safety checklist conflicts with V2 request publishing.**
   V2 prompt template requires a published V2 request report link, and the README says runtime artifacts such as V2 request report and generated prompt may live in `review/v2/...` branch. ([GitHub][14]) But the safety checklist says "Нет raw V2 request/response artifacts в snapshot", which can be read as "do not include the V2 request report in the review snapshot." ([GitHub][17])
   This is the main practical contradiction. It should say: V2 request report / generated prompt / safety report are allowed in the **review branch after approval**, but raw response / ingest artifacts are not allowed in `main` by default.

2. **Leftover `recording step` / `interpretation step` wording still creates ambiguity.**
   README status table says `raw_response_captured` means "recording step" and `ingested` means "interpretation step"; response template also says next step is "interpretation step." ([GitHub][11]) At the same time, README and Kilo mode contract clearly say V2 does not use `kilo-recorder` and ingest is manual through ordinary Kilo runs. ([GitHub][11])
   This is not fatal, but the wording should be changed to avoid any mental link to `kilo-recorder`.

3. **V2 prompt points to central `/v1` rules without enough override text.**
   The prompt template tells the external chat to read central rules, but those central rules are explicitly a `/v1` prompt-only contract with required `External Question ID`, while V2 requires `V2 ID` and a different answer shape. ([GitHub][1])
   This can be fixed by adding one sentence: "Use central rules for honesty, readback, source boundaries and no-local-repo-claims; V2-specific response structure in this prompt overrides `/v1`-specific section names."

4. **Runtime V2 directories are not fully normalized across docs.**
   README says runtime artifacts may exist, response template points to `.ai/external_reviews/responses/...`, request template references `.ai/external_reviews/requests/...`, but the README required structure only lists README, navigation and templates. ([GitHub][11])
   This is not a conceptual conflict, but it can create practical confusion: where exactly do requests, prompts, safety reports, responses, ingest summaries and sanitized summaries live, and which are tracked in `main`?

5. **`V2_navigation.md` has a `Request Path`, but request reports may not exist in `main`.**
   V2 navigation is tracked in `main`, but request reports may live only in a review branch or local-only. ([GitHub][12]) If `Request Path` points to a file not present in `main`, future readers may see a broken or misleading path. Add branch/commit columns or rename to `Request location`.

### Missing fields / missing docs

1. **No explicit sanitized summary template/path.**
   Docs say sanitized accepted summaries may be tracked in `main`, but there is no `v2_sanitized_summary_template.md` or clear directory like `.ai/external_reviews/summaries/`. ([GitHub][6])
   Either add a template, or explicitly say the ingest summary can be promoted only after sanitization and where it goes.

2. **Request template lacks a dedicated "links for external prompt" table.**
   It has base commit, snapshot commit, compare link and changed files, but the V2 prompt needs base file links and snapshot file links. ([GitHub][13]) Add a table: `File`, `Base raw URL`, `Snapshot raw URL`, `Why relevant`.

3. **Prompt template lacks project-context link slots.**
   For real senior review, the external chat often needs AGENTS, repo navigation, specific docs, bug journal or previous V1. The current prompt has central rules, request report, base/snapshot files and compare link, but no explicit `Required Project Context` or `Optional Project Context` section. ([GitHub][14])

4. **Safety checklist does not list `.ai/external_reviews/**` runtime paths clearly.**
   It blocks `.ai/external_chats/...` runtime paths, but not `.ai/external_reviews/requests`, `responses`, `prompts`, `safety`, `ingest`, or `summaries`. ([GitHub][17]) It has a generic V2-specific check, but path-level rules should be explicit.

5. **Cleanup could preserve evidence more clearly.**
   README cleanup says remove/keep branches, update navigation, status and draft PR state. ([GitHub][11]) It should also require preserving at least: V2 ID, base commit, snapshot commit, compare link, final status, and whether the review branch was deleted.

### Unsafe wording / ambiguity

1. **"snapshot" is used ambiguously.**
   Sometimes it means WIP code branch, sometimes a commit, sometimes the whole external review package. This causes the safety checklist contradiction. Use terms like `review branch snapshot`, `snapshot commit`, `main`, and `raw response artifacts` separately.

2. **"raw V2 request/response artifacts" mixes two different risk classes.**
   Request report and prompt may be intentionally published in the review branch so the external chat can work. Raw external response should not be published to `main` by default. They should not be grouped together with one rule. ([GitHub][11])

3. **`raw_response_captured` status still sounds recorder-like.**
   It is acceptable if documented, but I would rename or clarify it: `external_answer_saved` may be clearer than `raw_response_captured`.

4. **`zapushina` typo in README status table.**
   In `snapshot_pushed`, the Russian text says "запушина"; better "запушена". ([GitHub][11])

5. **Mojibake in `kilo_mode_contract.md`.**
   The "Repo-wide policy inheritance" section contains corrupted Russian text: `Р'РЎРµ ordinary Kilo modes...`. ([GitHub][9]) This should be fixed before verifier/pilot because it is a workflow contract file.

6. **`/v2` is placed in the short-entry-command area while saying it is not a shortcut like `/v1` or `/r1`.**
   `agent_protocol.md` says `/v2` is not a new shortcut at `/v1` or `/r1` level, but the heading itself appears in the shortcut-command section. ([GitHub][8]) This is manageable, but add one clarifying sentence: "`/v2` is command-like user shorthand for a manual Kilo protocol, not a Codex external route shortcut."

### Keep as-is

* Keep the core V2 separation from `/v1` and `/r1`. The README, external chat manual and `/r1` request template now agree that V2 is not prompt-only `/v1` and not `/r1` published-artifact route. ([GitHub][11])
* Keep "no new Kilo mode." This is consistent across README, agent protocol and Kilo mode contract. ([GitHub][11])
* Keep "no `kilo-recorder` / no `kilo-notebook` for V2 ingest." This is now clear in the main V2 docs and mode contract. ([GitHub][11])
* Keep mandatory `/v2 preview` before push and the concrete human push packet. This directly addresses the previous safety concern. ([GitHub][11])
* Keep commit-pinned links and base/snapshot separation. The prompt template correctly prioritizes commit-pinned links and compare context. ([GitHub][14])
* Keep the human-review and bug-report fields in ingest summary. They align with project policies requiring `Human Check` and `Баги и сложности`. ([GitHub][16])

### Must fix before verifier/pilot

1. Rewrite the safety checklist V2-specific line so it does not block the V2 request report from the review branch. Make the rule specifically about **not leaking raw response / ingest / private runtime artifacts into `main`**.

2. Replace or clarify "recording step" and "interpretation step" everywhere in V2 docs/templates. Suggested wording: `external answer saved manually` and `Kilo understanding summary`.

3. Add a V2-specific override note in `v2_prompt_template.md` explaining how central `/v1` rules should be used for V2.

4. Add explicit runtime directory/storage rules for:

   * `.ai/external_reviews/requests/`
   * `.ai/external_reviews/prompts/`
   * `.ai/external_reviews/safety/`
   * `.ai/external_reviews/responses/`
   * `.ai/external_reviews/ingest_summaries/`
   * `.ai/external_reviews/summaries/` or equivalent.

5. Add external review link table to the request template: base raw URL, snapshot raw URL, compare link and relevance.

6. Update `V2_navigation.md` so it can index review-branch/local-only artifacts without pretending they are normal `main` paths.

7. Fix the mojibake in `kilo_mode_contract.md`.

8. Add project-context link slots to the prompt template so Kilo can include AGENTS, repo navigation, bug journal, prior V1 or relevant docs when needed.

### Residual risks

Even after these fixes, V2 remains risky because it intentionally publishes WIP code to a public GitHub branch. The docs correctly warn that public branch push is publication, but the real protection will depend on Kilo's actual discipline during `/v2 preview` and the human's attention before approving push. ([GitHub][11])

The first pilot may expose manual friction: building raw links, maintaining base/snapshot pairs, keeping request/prompt/safety artifacts consistent, and cleaning review branches. The current "no helper script" decision is reasonable for the first version, but manual mistakes are still likely. ([GitHub][6])

External chat may still overclaim if the V2 prompt is not strict enough, especially because it is asked to read central `/v1` rules while operating in a V2 route. This is fixable with clearer V2-specific prompt wording. ([GitHub][1])

### Final verdict

`good to implement with small fixes`

The docs are now broadly coherent and much safer than the previous plan. They do **not** need another major redesign. But before verifier/pilot, fix the safety checklist contradiction around V2 request artifacts, clean up recorder-like wording, add runtime directory rules, improve V2 prompt/request templates, fix `V2_navigation.md` path semantics, and repair the mojibake in `kilo_mode_contract.md`.
