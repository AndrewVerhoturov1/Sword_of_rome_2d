The Block Orchestrator Package is a hard gate before any B1 execution begins. create_block_plan.md, create_block_orchestrator_package.md, review_block_report.md are the canonical prompts for these stages. ### Confirmed from provided excerpts

**What is strong in the current plan**

Correct macro-sequencing instinct. The plan correctly identifies that you need a Task Control Pack and Boss bootstrap before running any B1 block. This mirrors the hierarchy from the source idea: Task Control Pack → Boss Orchestrator Chat → B1 Block Chats.

Pilot-before-hardening gate exists. The plan does include a single B1 pilot before scaling. That's the right instinct — don't build the whole factory before proving one machine works.

Promotion to central core is deferred. Keeping central-core promotion last (Phase 4+) rather than front-loading it is the right call. Premature promotion of unproven patterns into central core is one of the classic failure modes.

Docs-only pilot is honest about its scope. Picking a docs-only block for the first pilot is a reasonable risk reduction — less surface area, no code execution risk, faster feedback.

The plan acknowledges the existing B1 scaffold. It doesn't pretend to start from scratch, which is correct given the audit finding that the scaffold already exists and is strong.

**Main findings / critique**

Finding 1: Phase ordering smuggles in full subproject architecture before the pilot proves anything.

Phase 1 of the plan builds the entire subproject file skeleton: main status, detailed plan, short working plan, block index, session journal, decision log, Boss bootstrap, handoff files. That is 8–10 canonical documents. Then Phase 2 runs a pilot. This is backwards. You are building the full bureaucratic superstructure of a subproject before you know if the B1 execution loop actually works end-to-end. If the pilot fails or reveals that the handoff format is wrong, you've already committed to a document architecture that may need restructuring. The pilot should drive the document design, not the other way around.

Specifically: Block Index, Decision Log, and Handoff для B1-блоков cannot be correctly designed without running at least one real block first. Their structure depends on what information actually flows between Boss and B1 during execution. Building them upfront is cargo-culting the templates without empirical evidence of what they need to contain.

Finding 2: The docs-only pilot is weak in a specific dangerous way.

"Docs-only" means: no game code touched, no integration test, no runtime behavior verified. The B1 scaffold (as the audit correctly noted) has never had a confirmed real run. A docs-only pilot exercises only the administrative part of the B1 loop (Plan → Package → Report → Review) but tells you nothing about whether the Block Orchestrator Package gate actually blocks wrong executions, whether the Junior Orchestrator role separation holds under real chat pressure, or whether the Block Report format captures the right information when something goes wrong mid-block.

The danger: You complete the docs-only pilot, declare the system proven, then promote it. But all you proved is that you can fill out forms. The first real code-affecting block will be the actual stress test, and it arrives after you've already committed to the architecture.

This is not an argument against a docs-only first pilot — it's an argument that the plan needs an explicit second pilot with non-trivial task scope before Phase 3 hardening. The plan has no such gate.

Finding 3: Dual ID model (BLOCK-001 + B1-001) is an unresolved collision.

The plan introduces this as a problem to resolve but does not actually resolve it. It defers the decision to "Phase 1 setup" without specifying the resolution rule. This is an early commitment problem: every template, every index entry, every cross-reference in the subproject documents will encode one ID format. If you choose wrong in Phase 1 and discover the collision in Phase 2 (during the pilot), you have to retroactively fix all the canonical documents you built in Phase 1.

The plan should either: (a) resolve this explicitly before Phase 1 begins, or (b) explicitly not build the Block Index until after the pilot, so the ID format can be validated against real usage first. Currently it does neither — it says "figure it out in Phase 1" while simultaneously committing to building Block Index in Phase 1.

From block_plan_template.md and block_orchestrator_package_template.md, the existing templates already use BLOCK-NNN style IDs. If the subproject adds B1-NNN style IDs as a parallel system, you now have two ID spaces that reference the same artifacts. This is not a minor naming issue — it affects cross-references in Block Reports, review logs, the Block Index, and any future automation.

Finding 4: The Block Orchestrator Package gate is mentioned but not enforced as a hard stop.

codex_orchestrator.md and create_block_orchestrator_package.md establish the Block Orchestrator Package as a mandatory gate: the B1 block cannot begin until the Package is approved. The rollout plan treats this gate as a step in a list, not as a hard checkpoint with explicit pass/fail criteria. There is no defined answer to: who reviews the Package, what constitutes approval, and what happens if the Package is rejected?

In practice, without an explicit "Boss reviews Package and signs off" moment before execution starts, this gate becomes ceremonial. Given that the audit found no evidence of a real B1 run, this gate has never been tested under real conditions. A weak gate here means the pilot doesn't actually prove the system works — it just proves you can generate documents.

Finding 5: Missing prerequisite — Boss Orchestrator Chat setup is underspecified.

The plan says "create Boss Orchestrator Chat, bootstrap it." But codex_orchestrator.md specifies that the Boss Orchestrator operates in a persistent chat with context of the full subproject. The bootstrap requires: subproject context pack, Task Control Pack, current block index, current status. None of these exist before Phase 1, so you are bootstrapping the Boss with partial context. The plan does not address what the Boss does when asked to review a Block Orchestrator Package without having seen any prior execution history.

This is a real gap: the first time the Boss reviews a Package, it's operating cold with only template-level knowledge. The pilot may pass not because the system works but because there's nothing in the Package to reject yet (docs-only task has almost no failure modes).

Finding 6: Canonical docs vs. runtime/history artifacts are conflated in the document list.

The required subproject documents from the source idea include both canonical docs (Главный статус, Подробный план, Block Index) and runtime/history artifacts (Журнал сессии, Block reports, Decision log, Kilo reports). The plan builds all of them in Phase 1 as if they're all the same type of thing with the same creation timing. They are not:

- Canonical docs should exist before execution and define what execution looks like.
- Runtime artifacts should be created during and after execution and should never be pre-built with placeholder content.
- If you create a Журнал сессии or Decision log in Phase 1 before any session has occurred, you're either creating empty shells (which is bureaucracy without proof) or pre-filling them with hypothetical content (which is fiction dressed as history).

The plan has no explicit distinction between these two document categories and their timing.

Finding 7: "Hardening" phase (Phase 3) has no defined entry criteria.

The plan says: after the pilot, do hardening. But "hardening" means different things depending on what the pilot revealed. If the pilot showed that the Block Report template is missing a key field, hardening is template revision. If the pilot showed that the Boss/B1 role separation breaks down at the Package review stage, hardening is process revision. If the pilot worked perfectly (possible for a docs-only task), hardening might mean nothing at all.

Without explicit pilot review criteria that feed into hardening scope, Phase 3 is a blank check. The plan effectively says "do a pilot, then fix stuff, then scale." The "fix stuff" step is unstructured.

Finding 8: Promotion to central core has no trigger condition.

Phase 4+ says "promote reusable parts to central core after proof." But what constitutes proof? One successful docs-only pilot? Two pilots? One pilot with a code-affecting block? Without a specific trigger condition (e.g., "minimum 2 completed block cycles with reports reviewed and accepted by Boss, including at least one non-trivial task"), promotion could happen after the weakest possible evidence base.

Given that central core changes affect all future projects (not just this one), this trigger needs to be explicit and high-bar.

**Most dangerous assumptions**

- "Building the full subproject document architecture before the pilot proves it is safe." It isn't. If the pilot reveals structural problems, you've pre-built a document set against a wrong model.
- "A docs-only pilot is sufficient to validate the B1 execution loop." It validates the administrative forms. It does not validate the execution logic, role separation under pressure, or the Package gate's effectiveness.
- "The dual ID model can be resolved during Phase 1 setup." Without a defined resolution rule, this assumption hides a real commitment ambiguity that propagates through all documents.
- "The Boss Orchestrator can be effectively bootstrapped cold and immediately review a Block Orchestrator Package." The first Package review with no prior execution history is the weakest possible test of the review gate.
- "Phase 3 hardening will naturally address whatever the pilot revealed." Without explicit pilot exit criteria feeding into hardening scope, this is wishful thinking dressed as a plan step.

**What is missing before execution**

1. ID model resolution as a pre-Phase-1 decision, not a Phase-1 task. Write down the rule explicitly before creating any document.
2. Explicit categorization of canonical vs. runtime documents with different creation timing rules. Canonical docs: before execution. Runtime artifacts: during/after execution only.
3. Explicit Package gate pass/fail criteria — who approves, what signals rejection, what happens if rejected.
4. Boss bootstrap context specification — what context pack does the Boss receive at cold start, what is it allowed to be uncertain about.
5. Pilot exit criteria — specific questions the pilot must answer before Phase 3 begins. Not "did the pilot complete" but "what did it tell us about the loop."
6. Second pilot gate — a non-trivial (non-docs-only) second block cycle before Phase 3 hardening is declared complete. Otherwise the hardened system has only been tested on the easiest possible case.
7. Central core promotion trigger — specific, measurable conditions under which promotion is authorized.

**Minimal corrections**

These are not a new plan — just the minimum fixes to make the current plan executable without the identified failure modes:

1. Move ID resolution out of Phase 1 into a pre-Phase-1 decision gate. One paragraph, one rule, before any documents are created.
2. Split Phase 1 into two parts:
   - Phase 1a: Only canonical docs needed to start the pilot (Boss bootstrap, Task Control Pack, short working plan). Nothing else.
   - Phase 1b (after pilot): Full subproject document architecture, now informed by what the pilot revealed.
3. Add explicit pilot exit criteria as a checklist the Boss must sign off on before entering Phase 3. Minimum: Package gate was exercised, Report was filed, Review prompt was run, at least one finding or correction was identified (if the pilot produces zero findings, treat that as a signal the task was too trivial, not that the system is perfect).
4. Add a second pilot (Phase 2b) with a non-trivial task before Phase 3 hardening is locked. Docs-only is Phase 2a. A real block with a non-zero chance of mid-block correction is Phase 2b.
5. Add Boss bootstrap specification as a named artifact in Phase 1a — what exactly the Boss receives, in what format, at cold start.
6. Explicitly label runtime artifacts (Журнал сессии, Decision log, Block reports, Kilo reports) as created-during-execution-only, never pre-built.

**Would you still keep docs-only first pilot? Why or why not**

Yes, keep it — but reframe it honestly.

The docs-only pilot is valid as a forms validation exercise, not as a system validation exercise. It answers: "Can the team actually fill out the Block Plan, get a Package approved, file a Report, and run the Review prompt?" That is a real and valuable question, especially given that the audit found no evidence of any real B1 run ever.

But the plan currently treats the docs-only pilot as sufficient proof of the system. That framing is wrong and dangerous.

Keep the docs-only pilot as Phase 2a (administrative smoke test). Add Phase 2b as a non-trivial second block (real task, real decision pressure) before claiming the loop is validated.

If you skip the docs-only pilot entirely and jump straight to a real block, you risk two failure modes colliding simultaneously: administrative confusion (wrong template, unclear handoff) and execution confusion (wrong task scope, unclear role boundary). Separating them is smart. Just don't pretend Phase 2a is sufficient alone.
