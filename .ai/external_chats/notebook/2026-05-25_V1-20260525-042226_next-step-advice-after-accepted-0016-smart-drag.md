Play Sandbox Readiness Pack

Смысл: довести текущий игровой стол до состояния, когда он достаточно надёжен и понятен, чтобы стать preview-режимом для будущего редактора.

Состав шага:

```text
1. Stack/selection readiness:
   проверить и дополировать работу со стопками фишек.

2. Scenario reset:
   добавить простой сброс к исходному сценарию.

3. Basic data sanity check:
   добавить лёгкую проверку map/scenario/runtime references.

4. Final save/load regression check:
   убедиться, что save/load переживает move, drag, create, delete, control и reset behavior.
```

Это не новый большой roadmap, а один bounded readiness step.

### Why this step now

Сейчас уже есть много ручных возможностей: двигать, создавать, удалять, менять контроль, сохранять, загружать и перетаскивать. Следующая проблема уже не "добавить ещё одну кнопку", а убедиться, что всем этим можно пользоваться стабильно и понятно.

Этот шаг лучше, чем сразу начинать Map Editor, потому что будущий редактор должен запускать preview в уже достаточно надёжной песочнице. Если preview сам ещё хрупкий, редактор будет создавать данные, которые сложно проверять.

Этот шаг лучше, чем углублять rules engine, потому что цель сейчас — manual tabletop sandbox, а не строгая игра по правилам. `rulesHooks.ts` уже закрывает базовую validation/resolution, и строгие phase/combat/card rules сейчас преждевременны. ([GitHub][17])

Этот шаг лучше, чем продолжать save-system breadth, потому что `snapshot.ts` специально задуман как узкий one-slot runtime snapshot. Сейчас не нужны save slots, import/export, migrations or cloud persistence. ([GitHub][18])

### Near-term step sequence

#### 1. Play Sandbox Readiness Pack

Цель: закрыть минимальную готовность игрового стола перед редактором.

Внутри:

```text
- стопки фишек и выбор нужной фишки;
- выбранная фишка в стопке должна перетаскиваться именно как выбранная;
- сброс сценария к исходному fixture setup;
- лёгкая проверка данных map/scenario;
- финальная проверка save/load после всех ручных действий.
```

#### 2. Play Sandbox Acceptance / Cleanup Pass

Короткий стабилизационный шаг после readiness pack.

Внутри:

```text
- убрать явные UX-шероховатости;
- проверить, что старые действия не сломались;
- зафиксировать human check;
- записать повторяемые баги в bug journal, если они были.
```

Это не новый функциональный слой, а небольшой cleanup после насыщенного шага.

#### 3. Map Authoring 0.1 — First Editor Slice

После этого можно переходить к первому настоящему редакторскому шагу.

Минимальный смысл:

```text
- добавить точку;
- переместить точку;
- удалить точку;
- соединить две точки;
- получить map data, которую можно preview-нуть в Play Sandbox.
```

Без полноценного module editor, piece editor, cards, combat, Sword of Rome rules.

#### 4. Scenario Authoring 0.1 — Initial Setup Slice

После map editor base станет логичным следующий слой:

```text
- стартовая расстановка фишек;
- стартовый контроль точек;
- запуск preview из scenario setup.
```

Это уже связывает редактор с тем, что Play Sandbox умеет проверять.

### Branch points / sequencing risks

Главная развилка после `Play Sandbox Readiness Pack`:

```text
interaction depth vs authoring breadth
```

Если после readiness pack drag/selection/stack UX всё ещё неудобен, стоит сделать один короткий interaction cleanup. Если пользоваться столом уже удобно, надо идти в Map Authoring 0.1.

Второй риск — слишком рано начать "большой редактор всего". Правильнее начать с map editor only: spaces + connections. Piece editor, scenario editor и module packaging лучше не смешивать в первый editor slice.

Третий риск — scenario reset может случайно превратиться в save/load redesign. Этого делать не нужно. Reset должен просто восстановить исходный fixture-based scenario state, очистить runtime selection/messages/event log по понятному правилу и не менять canonical fixture files.

Четвёртый риск — stack handling может раздуться в полноценный stack manager. Пока достаточно минимального UX: видеть, сколько фишек на точке; циклически выбирать нужную; drag выбранной фишки из этой точки; не терять выбранную фишку после create/delete/load.

### What should wait

Сейчас не надо делать:

```text
- full rules engine;
- combat;
- cards/events subsystem;
- turn/phase strict automation;
- victory conditions;
- AI;
- network/multiplayer;
- mature save system;
- save slots/import/export/migrations;
- broad module package manager;
- full map editor + piece editor + scenario editor in one step;
- prototype migration;
- polished production UI.
```

Особенно важно: не превращать Play Sandbox в полноценную цифровую адаптацию конкретной игры. Его роль сейчас — быть ручным preview/runtime столом для будущего конструктора.
