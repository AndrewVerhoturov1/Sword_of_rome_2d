### Fixture expansion advice

Расширение tiny map до примерно 6 spaces стоит включить **в этот же следующий шаг**, но как жёстко ограниченный prep-substep.

Причина: на 2 spaces smart drag почти нечего проверять. Нужны разные случаи:

```text
space рядом и соединён
space рядом, но не соединён
space далеко
несколько возможных ближайших целей
цепочка connections
```

Но расширение fixture не должно превращаться в map editor work.

Рекомендованный объём:

```text
6 spaces
6–7 connections
одна стартовая фишка
существующие factions/pieceDefinitions без расширения
без zones
без terrain rules
без новых модулей
без красивой карты
```

То есть это не "дизайн карты", а тестовая схема для drag UX. Текущий visible fixture действительно содержит только два spaces и одну connection, поэтому расширение оправдано. ([GitHub][21])

Лучший формат: один handoff, но две внутренние части:

```text
Part A: expand tiny map fixture to 6 spaces.
Part B: implement smart drag move interaction.
```

Не делать отдельный большой prep-step, потому что тогда снова будет слишком мелкое движение. Не откладывать, потому что drag UX без нормальной карты плохо проверяется.

### Risks before implementation

1. **Phaser может случайно стать authority.**
   Во время drag Phaser неизбежно будет хранить временный visual drag state. Это нормально. Но Phaser не должен менять `GameState`, создавать committed events или решать окончательное перемещение.

2. **Drag state может смешаться с runtime state.**
   Временные значения вроде `draggedPieceId`, `pointerX`, `pointerY`, `snapTargetSpaceId`, `dragStartSpaceId` должны быть transient UI/input state. В `GameState` должен попасть только результат после committed `piece_moved`.

3. **Нужно не сломать existing runtime pipeline.**
   Сейчас pipeline в `App.tsx` уже проходит через `validateAction`/`resolveAction`, event commit, reducer и log. Drag должен заменить только способ ввода, а не всю цепочку. ([GitHub][12])

4. **Fixture expansion может расползтись.**
   6-space map должна быть простой тестовой схемой. Не нужно делать красивую Sword of Rome-like карту, terrain rules, zones или editor integration.

5. **Invalid drop должен быть понятным.**
   Если отпустить фишку не на target, или target найден, но RulesHooks блокирует ход, состояние не должно измениться. Пользователь должен увидеть простое сообщение, например: "Нельзя переместить фишку: между этими точками нет связи."

6. **Save/load edge case.**
   После drag move надо сохранить, обновить страницу, загрузить и убедиться, что фишка осталась в новой точке, event log восстановился, и следующий drag move продолжает создавать корректное событие.

7. **Pointer coordinate mismatch.**
   Phaser scale/fit может усложнить координаты pointer → world/canvas. Текущий `phaserScene.ts` уже использует `pointer.x/y` для hit-test по spaces; drag implementation должен остаться в той же координатной системе Phaser, не тащить screen/world conversion из React/SVG prototype напрямую. ([GitHub][15])

### Why this step now

Это лучше, чем prototype migration: reference prototype полезен ощущением drag, но архитектурно он другой, без Phaser и с прямым изменением UI state. Текущий проект уже имеет правильную runtime boundary, и её нужно сохранить. ([GitHub][28])
