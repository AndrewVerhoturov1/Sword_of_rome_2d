### Prototype usage strategy

Использовать prototype нужно в четырёх ролях.

#### 1. UX reference board

Смотреть, как там решены:

```text
левая панель объектов
контекстное меню на холсте
правый инспектор
режимы инструментов
валидация карты
русские подписи
```

Это reference for behavior and screen organization, not source of truth.

#### 2. Selective donor

Из prototype можно заимствовать идеи и отдельные маленькие utility-паттерны после review:

```text
как добавлять точку через контекстное меню
как выбирать объект
как показывать inspector
как показывать validation panel
как организовать список объектов
```

Но implementation должна быть переписана внутри текущей product architecture.

#### 3. Extraction source only for small isolated pieces

Не весь `App.jsx`, а только маленькие идеи:

```text
простая форма инспектора
список объектов
русские тексты
validation affordances
простые операции with spaces/connections
```

#### 4. Reference-only for advanced features

Слои, зоны, curve/Bezier connection points, rich styling, shadows, label styling, image layers, map rotation/scale — всё это полезно, но для первого editor slice должно остаться reference-only.

### What to borrow first

В первую очередь стоит взять не "фичи", а базовые редакторские взаимодействия:

```text
1. Контекстное меню "Добавить точку здесь".
2. Выбор точки на карте.
3. Перетаскивание точки.
4. Простая правая панель свойств выбранной точки.
5. Создание связи между двумя точками.
6. Выбор и удаление связи.
7. Левая панель списка spaces/connections.
8. Простая validation panel: ошибки и предупреждения.
```

Эти части напрямую соответствуют первому editor slice и видны в prototype через context menu, inspector and validation panel. ([GitHub][31])

### What not to copy directly

Нельзя переносить wholesale:

```text
1. Один большой App.jsx как архитектуру продукта.
2. Prototype state shape как canonical schema.
3. Prototype localStorage/save format как module package format.
4. Prototype renderer approach вместо текущей React + Phaser границы.
5. Слои/изображения/поворот/масштабирование карты в первый slice.
6. Зоны.
7. Кривые/Bezier точки связей.
8. Богатое оформление точек: тени, opacity, label fonts, colors.
9. Prototype validation как окончательный validator.
10. Любую логику, которая смешивает authoring state и runtime GameState.
```

Особенно важно: authoring state меняет `MapDefinition`; runtime state меняет `GameState`. Эти две вещи нельзя тихо смешивать. `module_product_ux_model.md` прямо фиксирует анти-смешивание authoring changes и runtime changes. ([GitHub][14])

### Recommended first editor slice

Рекомендованный bounded scope:

```text
Map Authoring 0.1 — Spaces and Connections Slice
```

Must-have:

```text
1. Новый явный режим "Редактор карты".
2. Отдельное authoring-состояние карты, не GameState.
3. Загрузка текущего tiny `map.json` в editable `MapDraft`.
4. Отображение spaces и connections в редакторе.
5. Добавить точку через контекстное меню или кнопку.
6. Выбрать точку.
7. Переместить точку drag-ом.
8. Удалить точку.
9. Переименовать точку.
10. Создать связь между двумя точками.
11. Удалить связь.
12. Простая левая панель: список точек и связей.
13. Простая правая панель: свойства выбранной точки/связи.
14. Простая проверка карты: broken connection refs, duplicate IDs, empty map warnings.
15. Кнопка "Проверить в песочнице".
```

Preview behavior:

```text
Редактор создаёт/обновляет MapDraft.
Preview создаёт временный GameState из MapDraft + minimal scenario setup.
Play Sandbox проверяет карту.
Runtime preview не должен silently write back в MapDraft.
```

Data boundary:

```text
MapDraft / MapDefinition:
- spaces
- connections
- coordinateSystem
- map metadata

GameState:
- pieces
- controlState
- eventLog
- runtime snapshot
```

### Scope cuts

Не включать в первый editor step:

```text
1. Слои изображений.
2. Зоны.
3. Кривые связи.
4. Поворот/масштабирование слоя карты.
5. Импорт пользовательских картинок.
6. Богатый стиль точек и линий.
7. Полный export/import module package.
8. Piece/Faction Editor.
9. Scenario Editor.
10. Cards.
11. Rules metadata editor.
12. Undo/redo.
13. Save-as-scenario from runtime.
14. Full validation framework.
15. Production UI polish.
```

Из prototype это всё выглядит полезным, но если включить сразу, editor branch станет mini-release, а не bounded slice.

### Integration with existing sandbox

Связка editor branch с play sandbox должна быть явной.

Правильная модель:

```text
Authoring Mode:
редактирует MapDraft / MapDefinition.

Play Sandbox Preview:
получает карту как input,
создаёт временный runtime GameState,
позволяет подвигать фишки и проверить связи.

Runtime Save:
сохраняет GameState + eventLog,
но не перезаписывает MapDraft.
```

Нельзя делать так:

```text
подвинул фишку в preview → это изменило scenario fixture
подвинул точку в editor → это изменило текущий runtime GameState
сохранил runtime snapshot → это стало map.json
```

Между режимами должен быть явный переход:

```text
Редактор карты → Проверить в песочнице → Вернуться в редактор
```

На первом шаге можно не делать полноценный файл-сейв. Достаточно, чтобы editor держал `MapDraft` в React state and preview used that draft. Но shape should stay close to canonical `map.json`.

### Near-term sequence

#### 1. Map Authoring 0.1 — Spaces and Connections Slice

Минимальный редактор карты: spaces, connections, инспектор, validation, preview. Это следующий главный шаг.

#### 2. Map Authoring 0.1 Hardening

После первого slice: исправить UX holes, broken refs, duplicate IDs, delete-space behavior, connection cleanup, preview failure messages.

#### 3. Scenario Authoring 0.1

Стартовая расстановка фишек и стартовый контроль точек поверх созданной карты.

```text
поставить стартовую фишку
выбрать owner
выбрать стартовый контроль space
preview scenario in sandbox
```

#### 4. Piece/Faction Authoring 0.1

Минимально редактировать стороны и типы фишек.

```text
faction id/name/color
piece definition id/name/owner/default count
```

#### 5. Module Workspace Shell

Связать карту, сценарий, фишки/стороны and preview into one explicit authoring workspace.

Это не полугодовой roadmap, а ближайшая локальная последовательность после того, как play sandbox уже готов.

### Risks / branch points

#### 1. Самый большой риск — смешать authoring и runtime

Редактор меняет definitions. Песочница меняет runtime state. Это главный boundary.

#### 2. Prototype может соблазнить на wholesale migration

Prototype кажется богатым, но это один большой React prototype без Phaser. Его нельзя делать foundation текущего product code. ([GitHub][30])

#### 3. Первый editor slice может расползтись

Если сразу взять layers, zones, labels, style, Bezier, import image, validation, preview, export, undo — задача станет слишком большой.

#### 4. Delete-space behavior требует аккуратности

Если удалить точку, надо решить, что делать со связями. Для first slice достаточно:

```text
delete space also removes connections touching that space
show simple warning/confirmation in Russian
```

Не нужен сложный migration flow.

#### 5. Preview needs minimal scenario fallback

Если editor map has no scenario pieces, preview должен либо создать temporary test piece on first available space, либо показать понятное сообщение. Лучше для первого slice: create a temporary preview scenario with one test piece if at least one space exists.

#### 6. ID policy

Технические IDs должны оставаться английскими: `space-1`, `connection-1`, not `точка-1`. Visible names may be Russian. AGENTS language policy already requires this split. ([GitHub][6])
