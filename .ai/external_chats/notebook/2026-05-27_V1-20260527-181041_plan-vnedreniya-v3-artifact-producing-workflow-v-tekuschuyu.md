# План внедрения V3 artifact-producing workflow в текущую систему repo

## 1. Главная поправка к V3: не фиксировать вечный запрет на product code

V3 core не должен содержать постоянный запрет на продуктовый код.

Правильная модель такая:

```text
V3 не решает глобально, можно ли трогать product code.
Каждый конкретный V3 request сам задаёт scope.
```

То есть в одном V3-запросе может быть:

```yaml
scope: docs_only
```

в другом:

```yaml
scope: scripts
```

в третьем:

```yaml
scope: product_code
```

и это нормальная часть системы.

V3 должен требовать не запрета, а явного описания:

```yaml
scope: product_code
allowed_paths:
  - "table-sandbox/src/"
forbidden_paths:
  - ".git/"
  - "_local/"
overwrite_policy: explicit
review_required:
  - codex
  - human
```

Решение, когда переходить к product code, принимает человек/Codex в конкретной задаче. Это не должно быть зашито как вечный запрет в V3-документах.

---

## 2. Что нужно взять за образец из V1 и V2

### Из V1 взять

V1 даёт модель лёгкого внешнего вопроса:

```text
Codex готовит prompt
человек копирует его во внешний чат
внешний чат отвечает
ответ сохраняется как notebook entry
V1_navigation получает краткую запись
```

Для V3 нужно сделать аналогичный слой, но не для обычного ответа, а для artifact-producing request.

То есть нужен свой prompt-шаблон:

```text
create_v3_request_prompt.md
```

Он должен помогать Codex готовить V3-запрос так же строго, как сейчас V1 template помогает готовить `/v1` prompt.

### Из V2 взять

V2 даёт модель более тяжёлого lifecycle:

```text
request
preview
human gate
external answer
ingest
implementation
cleanup
navigation update
```

Для V3 особенно важны:

```text
lifecycle index;
статусы;
строгие artifact classes;
human gates;
Codex review;
разделение stable docs и runtime artifacts;
не считать внешний ответ authority;
обновлять navigation на этапах процесса.
```

V3_navigation должен быть устроен как V2_navigation: живой операционный индекс, а не декоративная страница.

## Что должно быть в `contracts/v3_scope_contract.md`

Это один из главных файлов.

Он должен описывать не запреты, а типы scope.

Пример:

```yaml
scope_types:
  docs_only:
    description: "Документация, инструкции, markdown, templates."
  workflow_docs:
    description: "Workflow contracts, prompts, checklists."
  schemas:
    description: "JSON/YAML schemas, type drafts."
  scripts:
    description: "Dev scripts, validators, import helpers."
  fixtures:
    description: "Test fixtures and example package content."
  tests:
    description: "Test files."
  product_code:
    description: "Product source code."
  mixed:
    description: "Несколько типов изменений сразу."
```

Обязательная мысль:



# Какие документы не создавать сразу

Не создавать сразу:

```text
scripts/v3/apply_v3_package.py
```

Причина: apply — самый рискованный шаг. Сначала нужен staging/journal/review pipeline.

Не создавать сразу:

```text
автоматическую интеграцию с git commit/push
```

Причина: V2 тоже сохраняет human push gate и не автоматизирует публикацию без подтверждения. ([GitHub][5])

Не создавать сразу:

```text
сложные JSON Schema validators
```

Сначала достаточно YAML templates + простой validator.

Не переносить V3 в central core сразу. Сначала обкатать в этом repo, потом уже думать о generalization.

# MVP boundary

MVP V3 считается готовым, когда есть:

```text
1. .ai/v3/README.md
2. .ai/v3/V3_navigation.md
3. contracts:
   - process
   - roles
   - scope
   - package
   - manifest
   - journal
   - acceptance
4. templates:
   - request
   - manifest
   - journal
   - codex review
5. prompts:
   - create_v3_request_prompt.md
   - kilocode_notebook_v3_mode_prompt.md
6. manual setup guide для KiloCode UI
7. первый V3 request
8. первый external artifact package
9. первый Kilo journal
10. первый Codex review
11. первый human verdict
```

MVP не обязан иметь scripts. MVP не обязан иметь product_code pilot. Но MVP должен быть устроен так, чтобы product_code scope был возможен как явно выбранный режим в будущем.

# Главные риски

## Риск 1 — V3 начнёт подменять Codex

Нельзя, чтобы внешний ChatGPT сам решал, что принято в repo.

Решение:

```text
External ChatGPT creates package.
Codex reviews.
Human accepts.
```

## Риск 2 — Kilo начнёт быть смысловым инициатором

Kilo должен быть writer/executor, а не архитектор решения.

Решение:

```text
Codex prepares request.
Kilo imports/checks/writes/journals.
```

## Риск 3 — ZIP начнут считать доверенным

ZIP нельзя считать правдой сам по себе.

Решение:

```text
manifest + checksums + allowed paths + journal + Codex review.
```

## Риск 4 — V3_navigation станет декоративным

Нужно сразу сделать его lifecycle index, как V2_navigation.

Решение:

```text
обновлять V3_navigation на этапах:
request_prepared;
artifact_package_received;
staged;
journal_written;
codex_reviewed;
accepted/needs_revision/rejected.
```

## Риск 5 — product_code ошибочно запретят навсегда

Это была бы неправильная архитектура.

Решение:

```text
не forbidden;
а scope-controlled.
```

## Риск 6 — raw ZIP/staging попадут в public main

Нужно заранее решить storage policy.

Рекомендация:

```text
stable V3 docs/templates/contracts — tracked;
raw ZIP, external raw responses, staging dumps — local-only по умолчанию;
sanitized accepted summaries — только после Codex/human acceptance.
```

# Acceptance gates

Минимальные gates:

```text
1. Codex request gate
   Проверяет, что V3 request понятен, scope указан, paths заданы.

2. Human external-send gate
   Человек решает, отправлять ли request во внешний ChatGPT.

3. Artifact receive gate
   Человек скачивает ZIP и передаёт его в Kilo.

4. Kilo import gate
   Kilo проверяет manifest, checksums, paths, extra files, overwrite policy.

5. Journal gate
   Без journal V3 не считается выполненным.

6. Codex review gate
   Codex проверяет journal и реальные файлы.

7. Human meaning gate
   Человек подтверждает смысловой результат.

8. Navigation gate
   V3_navigation.md обновляется итоговым статусом.
```

# Итоговая формула

V3 нужно встроить как третий полноценный workflow route:

```text
V1 = внешний prompt-only анализ
V2 = внешний review конкретного snapshot
V3 = внешний artifact-producing package
```

Роли:

```text
Codex:
  ставит задачу, задаёт scope, проверяет результат

External ChatGPT:
  создаёт ZIP artifact package

Human:
  переносит request, скачивает ZIP, принимает финальное решение

KiloCode Notebook V3:
  проверяет package, пишет файлы, создаёт journal
```

Самое главное:

```text
V3 core не запрещает product_code.
V3 core требует явный scope, manifest, journal, Codex review и human acceptance.
```

Так V3 будет не хаотичной "нейросетью, которая пишет файлы", а нормальным workflow-слоем рядом с V1 и V2.
