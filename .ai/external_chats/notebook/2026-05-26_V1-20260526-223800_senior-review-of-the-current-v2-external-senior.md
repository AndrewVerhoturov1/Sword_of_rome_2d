### Confirmed from current project docs

Current V2 docs now define V2 as a manual protocol for external senior review of real WIP code through a pinned GitHub snapshot. It is explicitly separate from `/v1` and `/r1`, not a new Kilo mode, and external V2 answers are planning/debug input rather than repo authority. ([GitHub][8])

The core chain is now documented: after push, Kilo must verify commit-pinned links, create an instantiated V2 prompt, show it to the human, and only then move toward `waiting_external_answer`; `snapshot_pushed` without prompt creation is explicitly called a protocol violation. ([GitHub][8])

The V2 docs now define `/v2` as an interrupt-command during an active Kilo run, distinct from `blocked-v2-recommended`; explicit user `/v2` does not require three failed attempts or a full blocked report. ([GitHub][8])

The docs clearly state that `/v2 preview` is mandatory before any push, public `review/v2/...` push is publication, and the human push gate must show exact files, blocked files, local-only hits and a simple yes/no question. ([GitHub][8])

V2 ingest is documented as manual: either direct paste into an ordinary Kilo run or a local response file, without `kilo-recorder`; `kilo-notebook` remains `/v1-only`. ([GitHub][8])

Cleanup is now documented as Codex-owned post-accept cleanup: default policy is to delete local/remote `review/v2/...` branches, delete temporary V2 runtime artifacts, and update `V2_navigation.md` to `cleaned`; human override can mark `kept_by_decision` or `cleanup_pending`. ([GitHub][5])

### Main assessment

Текущая V2-система уже **практически пригодна для работы**, но я бы не называл её полностью "идеальной". Вердикт: **production-worthy with small corrections**.

Есть ручной V2-протокол, обязательный preview, жёсткий human push gate, commit-pinned thinking, разделение `/v1`/`/r1`/V2, manual ingest без recorder/notebook, interrupt-contract, YOLO stop-state и cleanup-модель. Это уже рабочая система. ([GitHub][8])

Почему всё ещё нужны small corrections: в текущем cleaned trace есть два слабых места — недоступные report-ссылки `0026/0027` и расхождение между обязательным ingest artifact в README и фактической записью в `V2_navigation.md`, где ingest summary для первого пилота не сохранён. 

### What is already strong

1. **V2 больше не выглядит как эксперимент без формы.**
   В README есть чёткий маршрут, статусы, push gate, ingest, cleanup, artifact classes и safety model. ([GitHub][8])

2. **Разделение с `/v1`, `/r1`, `kilo-recorder`, `kilo-notebook` теперь нормальное.**
   `kilo_mode_contract.md` прямо говорит, что V2 не новый Kilo mode, ingest ручной, `kilo-recorder` не используется, `kilo-notebook` остаётся `/v1-only`. ([GitHub][7])

3. **Human push gate стал достаточно строгим.**
   В V2 README теперь прямо указано, что перед push Kilo обязан показать конкретный пакет, включая changed files, untracked files, suspicious ignored files, blocked files, local-only hits and exact files proposed/excluded. ([GitHub][8])

4. **Post-push gap закрыт в документации.**
   Документация теперь прямо запрещает завершать run после push до создания instantiated prompt и показа его человеку. ([GitHub][8])

5. **`/v2` как interrupt теперь описан хорошо.**
   Это важно именно для реальной работы в YOLO/debug-flow: пользователь может явно вызвать `/v2`, Kilo ставит текущую задачу на паузу, фиксирует WIP state и переходит в preview. ([GitHub][8])

6. **YOLO + human gate теперь разведены.**
   README прямо говорит, что `awaiting_human_push_approval` — валидное состояние остановки даже в YOLO-режиме, и продолжение без human confirmation является нарушением протокола. ([GitHub][8])

7. **Cleanup-модель стала понятной.**
   AGENTS фиксирует Codex-owned cleanup, default deletion, and human override; master plan уточняет распределение ролей Kilo/Human/Codex. ([GitHub][5])

### Contract flaws / gaps

1. **Report links 0026/0027 сейчас ломаются.**
   Это самый заметный текущий flaw. Вопрос передал их как project-specific links, но raw/blob URLs вернули 404. Если эти reports должны быть evidence trail для первого реального V2, их отсутствие или неправильный путь надо поправить. Если они intentionally cleanup-deleted, тогда их не надо указывать как current accepted state links в будущих V1/V2 prompts. 

2. **Первый pilot зафиксирован как исключение из нового artifact contract.**
   README теперь говорит: ingest summary обязателен даже при direct paste. Но `V2_navigation.md` для первого пилота говорит: "not persisted as a separate artifact during the first pilot." Это можно принять как legacy exception, но надо явно пометить именно так, иначе выглядит как нарушение текущего контракта. ([GitHub][8])

3. **Master V2 plan слегка устарел относительно README.**
   Master plan всё ещё имеет статус `approved-for-docs-implementation` и в одном месте перечисляет старый компактный набор статусов без `prompt_ready`, `cleanup_pending`, `kept_by_decision`. README и `V2_navigation.md` уже содержат более свежую lifecycle-модель. Это не критично, если master plan считается историческим, но как "current contract" он уже уступает README. ([GitHub][10])

4. **Codex-owned cleanup нуждается в одном уточнении.**
   Сейчас docs говорят, что cleanup Codex-owned, но master plan also says Kilo prepares cleanup preview. Это нормально как hybrid model, но в кратких местах типа AGENTS можно добавить одну фразу: "Kilo может подготовить cleanup preview, но окончательный post-accept cleanup и checkpoint verification принадлежат Codex." Сейчас это понятно из master plan, но чуть менее очевидно из короткого описания в AGENTS. ([GitHub][10])

### Bookkeeping / lifecycle rough edges

1. **`V2_navigation.md` честно отражает cleaned state, но сохраняет "дырку" в ingest trace.**
   Там честно написано, что raw response не stored separately, а ingest не persisted as separate artifact. Это хорошо с точки зрения честности, но плохо как пример для будущих запусков. Следующий запуск уже должен соответствовать README: direct paste разрешён, но ingest summary всё равно должен существовать. ([GitHub][9])

2. **Request/Prompt locations после cleanup теряют проверяемость.**
   `V2_navigation.md` указывает request location as deleted branch and prompt location as deleted after cleanup. Это допустимо, но означает, что после cleanup будущий внешний читатель не сможет восстановить full request/prompt artifacts, если нет sanitized summary. Если traceability важнее минимализма, нужно сохранять sanitized summary. ([GitHub][9])

3. **Bug journal shortness.**
   Bug journal открылся как очень короткий файл. Если V2 pilot закрыл важный underlay/map-plane issue, стоит проверить, отражён ли итог в bug journal или project state. Я не могу утверждать, что это отсутствует локально, но по открытой версии bug journal не видно подробного закрытия. ([GitHub][12])

4. **Compare link usability remains brittle.**
   GitHub compare opened but failed to render full diff, showing "comparison is taking too long." V2 already mitigates this by requiring raw file links, but future prompts should not rely on compare as the only way to inspect changes. ([GitHub][17])

### Must fix before next real V2 run

1. **Resolve 0026/0027 report link issue.**
   Either restore/move these reports to the linked paths, or stop presenting them as readable current project links. Broken evidence links should not be part of a "current accepted state" review prompt.

2. **Mark the first pilot ingest gap as a legacy exception.**
   In `V2_navigation.md`, change the wording from just "not persisted as a separate artifact during the first pilot" to something like:

   ```text
   Legacy pilot exception before mandatory ingest artifact rule was fully enforced.
   Future V2 runs must persist ingest summary.
   ```

   This preserves honesty without weakening the current contract.

3. **Update master V2 plan status or mark it historical.**
   Either change status from `approved-for-docs-implementation` to something like `implemented/hardened`, or add a line: "Current operational contract is V2 README + V2_navigation; this master plan is historical planning context." This prevents agents from treating the older status list as current.

4. **Add one short cleanup ownership clarification to AGENTS or README.**
   Recommended wording:

   ```text
   Kilo may prepare cleanup preview and list branch/artifact state, but Codex owns final post-accept cleanup review, deletion/checkpoint, and verification that V2_navigation reflects final state.
   ```

5. **For the next real V2 run, enforce ingest summary strictly.**
   Direct paste is fine, but there must be an ingest summary artifact or an explicit accepted exception. The README already says ingest summary is mandatory. ([GitHub][8])

### Nice to improve

1. Add one small "ideal successful V2 run" example row or mini-scenario in README. This would make the process easier for Kilo and the user to follow.

2. Add a short "after cleanup, what remains in main?" section:

   V2_navigation row remains.

5. Consider a future helper script only for read-only checks/link generation, not for push or cleanup. Current manual process is acceptable, but raw-link generation and artifact inventory are easy places for human/model mistakes.

### Final verdict

`production-worthy with small corrections`

V2 is now usable as a real workflow. It has already succeeded once technically, and the current docs cover the main dangerous areas: manual preview, human push approval, public-branch warning, pinned snapshot, manual ingest, interrupt mode, YOLO stop-state and Codex-owned cleanup.

Before the next real V2 run, fix the broken report links or stop relying on them, mark the first pilot's missing ingest artifact as a legacy exception, clarify master-plan vs current-operational-contract status, and make sure future direct-paste ingest always produces an ingest summary. These are small corrections, not a redesign.
